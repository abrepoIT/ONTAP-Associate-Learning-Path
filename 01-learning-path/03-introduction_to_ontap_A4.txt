INTRODUCTION TO ONTAP

MODULE 1 - NETAPP ONTAP DATA MANAGEMENT SOFTWARE
==================================================

1.1 Cos'è ONTAP Software?
--------------------------
NetApp ONTAP (originariamente chiamato Data ONTAP) è il flagship data management operating system di
NetApp — il software che gira sui sistemi di storage NetApp e controlla come i dati vengono
memorizzati, acceduti, protetti e gestiti.

ONTAP non è semplicemente un sistema operativo di storage nel senso tradizionale. È una piattaforma
completa di data management che astrae l'hardware sottostante e presenta uno strato unificato e
intelligente per gestire i dati indipendentemente da dove risiedono — on-premises, nel cloud, o
entrambi.

Filosofia core di ONTAP:
ONTAP è costruito attorno a tre principi guida che attraversano tutte le sue funzionalità:

  SMART             POWERFUL              TRUSTED
  Semplifica        Risponde ai           Protegge e
  le operazioni     bisogni mutevoli      mette in sicurezza i dati

  Automazione       Performance Flash     Protezione Ransomware
  Efficienza        Scalabilità           Cifratura
  Risparmio costi   Hybrid Cloud          Compliance
  AI-driven         Consistenza

Breve storia di ONTAP:
  Data ONTAP 1.0                - 1992         - Prima release; OS fondamentale di NetApp
  Data ONTAP 7-Mode             - Anni 2000    - OS enterprise storage dominante; modello a controller singolo
  Clustered Data ONTAP (cDOT)   - 2011         - Introduzione del clustering scale-out; operazioni non distruttive
  ONTAP 9                       - 2016         - Rinominato "ONTAP 9"; focus unificato on-prem e cloud
  ONTAP 9.x                     - 2017-oggi    - Release continue con integrazione cloud, NVMe, funzionalità AI-driven
  ONTAP One                     - 2023         - Modello di licenza all-inclusive; tutte le funzionalità ONTAP in una licenza


1.2 ONTAP: Deployment Flessibile su Architetture Diverse
---------------------------------------------------------
Uno dei differenziatori più importanti di ONTAP è che gira su un'ampia varietà di modelli di
deployment, consentendo alle organizzazioni di usare lo stesso software, strumenti, interfacce di
gestione e procedure operative indipendentemente dall'infrastruttura sottostante.

Opzioni di deployment ONTAP:

NetApp AFF (All Flash FAS):
  - Piattaforme hardware All-NVMe o All-SSD
  - Progettato per le massime performance: latenza sub-millisecondo, milioni di IOPS
  - Esegue ONTAP nativo
  - Ideale per: database Tier-1, VDI, AI/ML, applicazioni mission-critical

NetApp FAS (Fabric-Attached Storage):
  - Array ibridi: mix di SSD e HDD
  - Performance e capacità bilanciate a costo inferiore rispetto agli AFA
  - Esegue ONTAP nativo
  - Ideale per: workload misti, storage secondario, ambienti NAS

NetApp ASA (All SAN Array):
  - Ottimizzato specificamente per workload block storage (SAN)
  - Design all-active per massime performance e disponibilità SAN
  - Esegue ONTAP nativo
  - Ideale per: workload SAN mission-critical, Oracle, SAP HANA

ONTAP Select:
  - ONTAP deployato come appliance virtuale software-defined su server x86 commodity
  - Gira in ambienti hypervisor VMware vSphere, KVM
  - Ideale per: uffici remoti/branch (ROBO), deploy edge, ambienti cost-sensitive

Cloud Volumes ONTAP (CVO):
  - ONTAP deployato come macchina virtuale cloud-native nei cloud pubblici
  - Disponibile su AWS, Azure e Google Cloud
  - Set completo di funzionalità ONTAP nel cloud
  - Ideale per: NAS/SAN cloud-native, tiering dati verso il cloud, servizi dati hybrid cloud

Amazon FSx for NetApp ONTAP:
  - Servizio ONTAP completamente gestito nativamente in AWS
  - Gestito da AWS, alimentato da ONTAP
  - Ideale per: workload AWS che richiedono NAS enterprise (NFS/SMB), accesso immediato a tutte le
    funzionalità ONTAP senza gestire l'infrastruttura

Azure NetApp Files:
  - Servizio file NFS/SMB enterprise-grade completamente gestito in Azure
  - Alimentato dalla tecnologia ONTAP
  - Ideale per: workload SAP, HPC, VDI, DevOps in Azure

ONTAP on Google Cloud (Google Cloud NetApp Volumes):
  - Servizio storage NetApp gestito in Google Cloud
  - Ideale per: workload Google Cloud che richiedono file storage enterprise

Il valore di una piattaforma unificata:
Eseguire lo stesso software ONTAP su tutti gli ambienti fornisce vantaggi chiave:

  Operazioni consistenti   - Stessa CLI, API e GUI su tutte le piattaforme — nessuna riqualificazione necessaria
  Gestione dati unificata  - Sposta i dati tra on-prem e cloud senza cambiamenti di formato
  Protezione dati portabile - SnapMirror replica tra qualsiasi istanza ONTAP, indipendentemente dalla posizione
  Policy di sicurezza uniformi - Cifratura, controllo accessi e audit log funzionano allo stesso modo ovunque
  Portabilità licenze      - Le licenze ONTAP One funzionano su on-prem e cloud

2.3 Requisiti di Storage Efficiency
-------------------------------------
Con la crescita dei dati, le organizzazioni sono sotto pressione per:
  1. Ridurre il footprint di storage fisico — acquistare meno drive, usare meno spazio rack,
     consumare meno energia
  2. Ridurre il costo per TB utilizzabile — massimizzare il valore di ogni euro speso in storage
  3. Gestire la data sprawl — prevenire la crescita incontrollata dal consumare i budget

I sistemi di storage tradizionali memorizzano i dati così come sono: un dataset da 10 TB richiede 10
TB di storage raw (più l'overhead RAID). ONTAP cambia fondamentalmente questa equazione tramite una
suite di tecnologie di storage efficiency che riducono la capacità fisica richiesta per memorizzare
la stessa quantità di dati.


2.4 Migliorare la Storage Efficiency con le funzionalità ONTAP
----------------------------------------------------------------

DEDUPLICATION
La deduplication elimina i blocchi di dati duplicati su un volume o aggregate.

Come funziona:
  1. ONTAP calcola un'impronta digitale (hash) per ogni blocco di dati
  2. Se due blocchi hanno la stessa impronta, viene conservata solo una copia fisica
  3. Tutti i riferimenti puntano allo stesso blocco fisico (approccio pointer-based)

Tipi:
  - Deduplication inline: gira in tempo reale mentre i dati vengono scritti (zero impatto
    sulle performance su AFF)
  - Deduplication post-process: gira in background dopo che i dati sono stati scritti
  - Deduplication cross-volume: deduplica su più volumi nello stesso aggregate
  - Deduplication a livello di aggregate (AGGR-dedup): deduplica i dati su tutti i volumi
    di un aggregate

Risparmio tipico: 2:1 fino a 5:1 per ambienti VDI, database e workload virtualizzati

COMPRESSION
La compressione riduce la dimensione dei dati codificandoli in modo più efficiente.

Tipi di compressione ONTAP:
  Inline compression         - Comprime i dati nel write buffer prima di scrivere su disco; applicata al momento della scrittura; ideale per workload I/O random
  Post-process compression   - Comprime i dati già memorizzati su disco; applicata in background; ideale per dati sequenziali/archivio
  Adaptive compression       - Seleziona automaticamente la dimensione del blocco in base al pattern dei dati; applicata al momento della scrittura; ideale per workload misti
  TSSE                       - Comprime aggressivamente i dati "freddi" (acceduti raramente); applicata in background; ideale per dataset grandi con data aging

Risparmio tipico: 1,5:1 fino a 4:1 a seconda del tipo di dati

COMPACTION
La compaction impacchetta più blocchi di dati piccoli in un singolo blocco di storage da 4KB.

Problema che risolve: molte applicazioni scrivono dati in piccoli chunk (512 byte, 1KB, 2KB). Senza
compaction, ogni piccola scrittura occupa un blocco completo da 4KB, sprecando la maggior parte
dello spazio allocato. ONTAP impacchetta i blocchi piccoli che insieme sommano meno di 4KB in un
singolo blocco fisico da 4KB, massimizzando l'utilizzo dello spazio disco.

THIN PROVISIONING
Il thin provisioning consente di allocare volumi con una dimensione logica maggiore dello spazio
fisico attualmente disponibile.

Esempio:
  - Un volume viene presentato all'host come 10 TB
  - Solo 2 TB di storage fisico sono effettivamente consumati (in base ai dati attuali)
  - Lo storage fisico viene allocato dinamicamente man mano che i dati vengono scritti

Vantaggi:
  - Rimandare gli acquisti di storage fino a quando la capacità è effettivamente necessaria
  - Evitare il sovra-provisioning "per sicurezza"

Rischio: se il consumo reale si avvicina alla capacità fisica senza aggiungere storage, i volumi
possono esaurire lo spazio. Richiede monitoraggio attivo.

FLEXCLONE
FlexClone crea cloni istantanei e space-efficient di volumi, file o LUN.

Come funziona:
  - Usa un meccanismo di copia pointer-based (redirect-on-write)
  - Il clone viene creato istantaneamente indipendentemente dalla dimensione del dataset
  - Inizialmente consuma zero spazio aggiuntivo (condivide tutti i blocchi con il parent)
  - Lo spazio viene consumato solo quando i dati nel clone divergono dal parent

Casi d'uso:
  - Sviluppo/test: clona istantaneamente database di produzione per i test (senza copiare dati)
  - DevOps: provisioning di ambienti isolati in secondi
  - VDI: clona immagini master per centinaia di desktop
  - Data analytics: clona i dati di produzione per l'analisi senza impattare le performance

Riepilogo Storage Efficiency ONTAP:
  Deduplication    - Elimina: blocchi duplicati         - Risparmio: 2:1 - 10:1
  Compression      - Elimina: dimensione dei dati       - Risparmio: 1,5:1 - 4:1
  Compaction       - Elimina: spreco I/O piccoli        - Risparmio: 1,2:1 - 2:1
  Thin Provisioning - Elimina: capacità sovra-allocata  - Dipende da allocazione vs uso reale
  FlexClone        - Elimina: overhead cloni            - Quasi zero per i cloni
  Combinato (AFF)  - Tutto quanto sopra                 - 4:1 - 10:1+ effettivo

NetApp garantisce un minimo di efficienza storage 4:1 sui sistemi AFF o fornisce un credito di
storage. Molti deployment reali raggiungono rapporti significativamente più alti.


2.5 ONTAP Features: FabricPool — Tiering Automatico dei Dati
--------------------------------------------------------------
FabricPool sposta automaticamente i dati "freddi" (acceduti raramente) dallo storage flash costoso
verso object storage a basso costo (cloud o on-premises), mantenendo i dati caldi sul flash.

Come funziona:
  1. ONTAP monitora la temperatura dei blocchi (quanto frequentemente viene acceduto ogni blocco)
  2. I blocchi non acceduti per un periodo configurabile (cooling period) vengono taggati come
     "freddi"
  3. I blocchi freddi vengono automaticamente tiered verso l'object store collegato (AWS S3, Azure
     Blob, GCP GCS, NetApp StorageGRID)
  4. Quando i dati freddi vengono nuovamente acceduti, vengono recuperati trasparentemente e
     possono essere promossi di nuovo sul flash

Policy di tiering:
  Snapshot-only - Solo i dati negli snapshot non presenti nel filesystem attivo vengono tiered
  Auto          - I dati freddi sia nel filesystem attivo che negli snapshot vengono tiered
  All           - Tutti i dati vengono spostati immediatamente al cloud tier (utile per l'archiviazione)
  None          - Nessun dato viene tiered; tutti i dati rimangono sul flash locale

Vantaggi:
  - Ridurre i costi dello storage flash del 30-70% per workload con dati freddi
  - Mantenere gli SLA di performance per i dati caldi (sempre sul flash)
  - Trasparente per le applicazioni — nessuna modifica richiesta

3.1 Requisiti per l'agilità aziendale
---------------------------------------
Le organizzazioni moderne hanno bisogno di un'infrastruttura storage che possa:
  - Scalare istantaneamente con il cambiare delle esigenze aziendali — senza interruzioni
  - Spostare i dati liberamente tra ambienti on-premises e cloud
  - Supportare workload diversi — block, file e object — su un'unica piattaforma
  - Mantenere le performance anche con la crescita della capacità
  - Abilitare l'accesso globale ai dati per team e applicazioni distribuite

ONTAP affronta tutti questi requisiti attraverso una combinazione di scelte architetturali e
funzionalità software.


3.2 Performance ottimizzate per lo storage Flash con ONTAP
-----------------------------------------------------------
I workload enterprise moderni — in particolare database, analytics, AI/ML e VDI — richiedono latenza
estremamente bassa e throughput elevato. ONTAP è architetturato per sfruttare appieno le capacità
dello storage flash NVMe.

Supporto NVMe e NVMe/oF:
ONTAP supporta lo stack NVMe completo, sia per lo storage interno che per la connettività host.

NVMe per storage interno:
  - ONTAP rileva i drive NVMe e li usa nativamente tramite il protocollo NVMe
  - Bypassa i layer di traduzione SCSI legacy per il massimo throughput
  - Riduce drasticamente la latenza dello storage stack

NVMe over Fabrics (NVMe-oF) per la connettività host:
  - NVMe/FC (Fibre Channel): protocollo NVMe su fabric Fibre Channel — latenza sub-100µs
  - NVMe/TCP: NVMe su Ethernet TCP/IP standard — più semplice da deployare, buone performance

Quality of Service (QoS):
Le policy QoS in ONTAP consentono agli amministratori di controllare e dare priorità agli I/O tra i
workload.

  Maximum QoS (ceiling) - Limita il throughput o gli IOPS massimi di un workload per evitare
                          che impatti gli altri
  Minimum QoS (floor)   - Garantisce a un workload un livello minimo di performance, anche sotto
                          carico
  Adaptive QoS          - Regola automaticamente le impostazioni QoS in base alla dimensione del
                          volume (scala con i dati)

Casi d'uso:
  - Proteggere un database Tier-1 dall'essere penalizzato da un job di backup in esecuzione sullo
    stesso sistema
  - Garantire IOPS minimi per i desktop VDI durante i periodi di picco
  - Applicare performance fair-share tra tenant in un ambiente multi-tenant

Storage Virtual Machine (SVM) — Multi-Tenancy:
Le SVM (dette anche Vserver) sono il meccanismo logico di multi-tenancy in ONTAP. Ogni SVM è
un'entità di storage logica completa e isolata con:
  - Il proprio namespace (albero di directory)
  - Le proprie interfacce di rete (indirizzi IP, FC WWPN)
  - I propri protocolli (NFS, SMB, iSCSI, FC, NVMe/FC — mix a piacere)
  - I propri utenti, gruppi e controlli di accesso
  - I propri volumi e dati

Vantaggi delle SVM:
  - Isolamento: i dati e la configurazione di una SVM sono completamente isolati da un'altra
  - Sicurezza: la compromissione di una SVM non influisce sulle altre
  - Mobilità: un'intera SVM può essere migrata tra cluster ONTAP in modo non distruttivo
  - Multi-tenancy: diverse business unit, applicazioni o clienti possono condividere
    l'infrastruttura fisica rimanendo logicamente isolati


3.3 Performance ottimali con ONTAP
------------------------------------

Architettura del cluster ONTAP:
Un cluster ONTAP è composto da uno a 24 nodi (a seconda della piattaforma), ciascuno dei quali
contribuisce con CPU, memoria e drive a un pool condiviso.

Componenti architetturali chiave:
  Node                - Un singolo storage controller (fisico o virtuale) che esegue ONTAP
  HA Pair             - Due nodi che si replicano reciprocamente la cache NVRAM e possono farsi
                        carico l'uno dell'altro
  Cluster Interconnect - Rete privata ad alta velocità (tipicamente 10/25/40 GbE) per la
                         comunicazione inter-nodo
  Aggregate           - Un RAID group di drive fisici assegnato a un nodo
  Volume              - Un contenitore dati logico all'interno di un aggregate (equivalente a un
                        filesystem)
  SVM                 - Un tenant logico / data server che possiede volumi e li presenta ai client

FlexVol e FlexGroup Volumes:

FlexVol:
  - Volume ONTAP standard
  - Risiede interamente in un aggregate (un nodo)
  - Dimensione massima: 100 TB (o 300 TB con large volumes abilitati)
  - Tempo di risposta sub-millisecondo

FlexGroup:
  - Un volume distribuito che si estende su più aggregate e più nodi
  - Scala fino a 20 PB e 400 miliardi di file
  - Parallelizza le operazioni di metadata e dati su tutti i volumi membro
  - Ideale per: grandi repository di dati non strutturati, dataset di training AI/ML, workflow
    media, HPC

Non-Disruptive Operations (NDO):
Una delle capacità più preziose di ONTAP: la possibilità di eseguire operazioni infrastrutturali
maggiori senza interrompere l'accesso delle applicazioni ai dati.

  Upgrade controller   - Storage tradizionale: downtime pianificato richiesto
                         ONTAP NDO: migrazione live al nuovo nodo, zero downtime
  Sostituzione drive   - Storage tradizionale: hot-swap (se esiste ridondanza RAID)
                         ONTAP NDO: hot-swap con rebuild RAID automatico
  Spostamento volume   - Storage tradizionale: copia + downtime
                         ONTAP NDO: spostamento volume non distruttivo (LIF segue i dati)
  Espansione cluster   - Storage tradizionale: complessa, spesso distruttiva
                         ONTAP NDO: aggiunta nodi mentre il cluster è in esecuzione
  Modifiche protocollo - Storage tradizionale: downtime per riconfigurazione
                         ONTAP NDO: modifiche protocollo online

LIF Mobility e Load Balancing:
Le LIF (Logical Interface) sono gli endpoint di rete (indirizzi IP o FC WWPN) tramite cui i client
accedono ai dati. Le LIF ONTAP possono migrare automaticamente tra porte fisiche in risposta a:
guasto di porta o nodo (failover automatico), bilanciamento del carico manuale, spostamenti di
volume non distruttivi. Questo garantisce che i client abbiano sempre un percorso di rete verso i
propri dati, anche durante la manutenzione hardware o i guasti.


3.4 Rispondere alle esigenze di storage in evoluzione con ONTAP
----------------------------------------------------------------

Supporto Multi-Protocollo:
ONTAP supporta in modo unico tutti i principali protocolli di storage simultaneamente sulla stessa
piattaforma:

  NFS v3/v4/v4.1   - File (NAS)   - File sharing Linux/Unix
  SMB 2.0/3.0/3.1.1 - File (NAS)  - File sharing Windows
  iSCSI             - Block (SAN)  - SAN su Ethernet
  Fibre Channel (FC) - Block (SAN) - SAN enterprise
  NVMe/FC           - Block (SAN)  - SAN a latenza ultra-bassa
  NVMe/TCP          - Block (SAN)  - NVMe SAN su Ethernet
  S3                - Object       - Accesso object cloud-native

Perché il multi-protocollo è importante: un singolo sistema ONTAP può servire simultaneamente VM
Linux via NFS, utenti Windows via SMB, database Oracle via iSCSI o FC, e applicazioni cloud-native
via S3 — tutto gestito da un'unica interfaccia.

FlexCache — Caching Distribuita:
FlexCache crea volumi di read-cache in posizioni remote che contengono un sottoinsieme di dati da un
volume di origine.

Come funziona:
  - Un volume FlexCache viene creato in un sito remoto (o cluster diverso)
  - Le richieste di lettura vengono servite localmente dal FlexCache (bassa latenza)
  - I cache miss vengono recuperati trasparentemente dal volume di origine
  - Le scritture possono essere dirette di nuovo all'origine (write-back) o gestite localmente

Casi d'uso:
  - Distribuzione file globale: team di ingegneria in più sedi accedono a file CAD/design grandi;
    FlexCache li serve localmente
  - Collaborazione multi-sito: riduzione della latenza per utenti che accedono a repository di
    dati centrali
  - Workload read-intensive: analytics, rendering, media processing

ONTAP S3 (Object Storage):
ONTAP include un servizio di object storage nativo compatibile con S3. Le applicazioni possono
accedere ai dati ONTAP tramite API S3 standard, abilitando applicazioni cloud-native ad usare
storage ONTAP on-premises. I bucket S3 possono essere supportati da volumi FlexVol o FlexGroup, con
accesso S3 multi-tenant tramite SVM.


3.5 ONTAP abilita l'agilità aziendale
---------------------------------------

SnapMirror — Mobilità dei dati e Replication:
SnapMirror è il motore di replica di ONTAP. Replica i dati tra volumi ONTAP — on-premises verso on-
premises, on-premises verso cloud, o cloud verso cloud.

Modalità SnapMirror:
  Async SnapMirror      - Replica gli snapshot periodicamente - RPO: Minuti - Uso: DR, backup offload, cloud tiering
  Sync SnapMirror       - Ogni scrittura confermata su primario e secondario - RPO: Zero - Uso: DR senza perdita dati
  SnapMirror active sync - Entrambi i siti servono I/O simultaneamente - RPO: Zero - Uso: Business continuity, failover di sito
  SnapMirror Cloud      - Replica verso object storage S3-compatible - RPO: Ore - Uso: Backup a lungo termine su cloud

SnapMirror Business Continuity (SM-BC) / Active Sync:
  - Consente a due sistemi ONTAP (o due istanze cloud) di servire entrambi I/O per lo stesso
    dataset
  - Se un sito si guasta, l'altro continua a servire I/O in modo trasparente
  - Le applicazioni non vedono alcuna interruzione — anche durante un guasto completo del sito
  - Usa l'ONTAP Mediator per il quorum di failover automatico

Gestione Hybrid Cloud Unificata con BlueXP:
NetApp BlueXP fornisce una singola vista operativa su:
  - ONTAP on-premises (AFF, FAS, ASA, ONTAP Select)
  - Cloud Volumes ONTAP (AWS, Azure, GCP)
  - Amazon FSx for NetApp ONTAP
  - Azure NetApp Files
  - Google Cloud NetApp Volumes

Servizi BlueXP:
  - BlueXP Backup and Recovery: backup cloud-integrato per ONTAP

4.1 Requisiti di Data Protection
----------------------------------
Le organizzazioni affrontano un insieme crescente e sempre più complesso di sfide di protezione
dei dati:

  Ransomware          - I criminali informatici cifrano i dati e richiedono un pagamento per le
                        chiavi di decifratura
  Cancellazione accidentale - L'errore umano è una delle cause più comuni di perdita di dati
  Guasto hardware     - Guasti di disk, controller, o sito completo
  Corruzione          - Corruzione dati silenziosa, bug applicativi o problemi firmware di storage
  Disastro            - Disastri naturali, guasti elettrici o incendi che distruggono un data center
  Compliance          - I requisiti normativi impongono retention, immutabilità e verificabilità

La regola 3-2-1-1-0 (3 copie, 2 supporti, 1 offsite, 1 air-gapped/immutabile, 0 backup non testati)
è la baseline moderna per la protezione completa dei dati — e ONTAP è progettato per soddisfare ogni
elemento di questo framework.


4.2 Funzionalità di Data Protection di ONTAP
---------------------------------------------

NetApp Snapshot Technology:
Gli Snapshot ONTAP sono la base di tutta la data protection in ONTAP.

Caratteristiche chiave:
  - Istantanei: uno snapshot di qualsiasi volume viene creato in millisecondi,
    indipendentemente dalla dimensione
  - Space-efficient: usa Copy-on-Write (COW) — inizialmente consuma zero spazio aggiuntivo
  - Application-consistent: ONTAP si integra con VMware VSS, Oracle RMAN e altre API di backup
    per creare snapshot crash-consistent o application-consistent
  - Non distruttivi: la creazione di snapshot non interrompe gli I/O delle applicazioni
  - Scalabili: fino a 1.023 snapshot per volume

Come funziona Copy-on-Write in ONTAP:
  Prima dello snapshot:   [Blocco A] [Blocco B] [Blocco C]  -- Filesystem attivo
  Dopo lo snapshot:       [Snap A]   [Snap B]   [Snap C]    -- Snapshot (stessi puntatori)
  Dopo modifica dati:     [Nuovo A]  [Blocco B] [Blocco C]  -- Filesystem attivo (A modificato)
                          [Vecchio A]    |           |
                              |      [Snap B]    [Snap C]   -- Snapshot conserva il vecchio A

Casi d'uso per il recovery:
  - Ripristino di un singolo file cancellato da un utente (secondi)
  - Ripristino di un intero volume a un punto nel tempo precedente (secondi o minuti)
  - Fornitura di un target di backup consistente a un'applicazione di backup
  - Creazione di un FlexClone per dev/test da uno snapshot di produzione

Snapshot Scheduling:
  - Snapshot orari, giornalieri e settimanali configurabili con retention automatica
  - Policy di esempio: mantieni 24 snapshot orari, 7 giornalieri, 4 settimanali

SNAPRESTORE
SnapRestore consente di ripristinare un intero volume o un singolo file/LUN da uno snapshot in
secondi — indipendentemente dalla dimensione del dataset.
  - Volume revert: ripristina un intero volume da 100 TB a uno snapshot precedente in meno di un
    minuto
  - Single file restore: ripristina singoli file senza ripristinare l'intero volume
  - LUN restore: ripristina un LUN specifico a uno stato precedente

SNAPMIRROR (Replica dei dati)
SnapMirror è centrale per la data protection:
  - Disaster Recovery: replica verso un sito secondario o cloud; failover se il primario si guasta
  - Backup offload: replica verso un sistema ONTAP secondario, poi esegui il backup da lì
    (rimuove il carico di backup dalla produzione)
  - Cloud backup: SnapMirror Cloud invia gli snapshot verso object storage S3-compatible per la
    retention a lungo termine

Workflow di recovery con SnapMirror:
  1. Il sito primario si guasta
  2. L'amministratore (o l'automazione) interrompe la relazione SnapMirror sul secondario
  3. Il volume secondario diventa read-write
  4. Gli host vengono reindirizzati al sito secondario
  5. Quando il primario viene ripristinato, il resync SnapMirror trasferisce solo i dati modificati

SNAPVAULT (ONTAP Backup)
SnapVault è la tecnologia di backup disk-to-disk di ONTAP, ottimizzata per la retention a lungo
termine delle copie di backup.
  - Trasferisce solo i blocchi modificati (non copie complete) — altamente efficiente
  - Mantiene un set separato di snapshot sul sistema secondario con policy di retention diverse
  - Il primario potrebbe mantenere 7 snapshot giornalieri; il secondario può mantenerne 12 mesi
    di mensili
  - Gli snapshot secondari sono indipendenti dal primario — la cancellazione degli snapshot
    primari non influisce sul secondario

METROCLUSTER
MetroCluster è la soluzione di mirroring sincrono hardware-based di NetApp per failover di sito a
zero RPO e zero RTO.

Configurazione:
  - Due cluster ONTAP in due siti (fino a 300 km di distanza per MetroCluster IP-based)
  - Tutti i dati vengono sincronicamente replicati al secondo sito in tempo reale
  - Se un sito si guasta completamente, il secondo sito prende automaticamente il controllo in
    secondi

Tipi di MetroCluster:
  MetroCluster FC - Distanza: fino a 300 km  - Connettività: Fibre Channel ISL
  MetroCluster IP - Distanza: fino a 700 km  - Connettività: Rete IP (con DWDM)

Casi d'uso:
  - Servizi bancari e finanziari (zero tolleranza per la perdita di dati)
  - Healthcare (i dati dei pazienti non possono andare persi)
  - Telecomunicazioni (disponibilità continua del servizio richiesta)


4.3 SnapLock — Storage Immutabile (WORM)
-----------------------------------------
SnapLock fornisce storage Write Once, Read Many (WORM) — i dati scritti su un volume SnapLock non
possono essere modificati o cancellati fino alla scadenza del periodo di retention.

Due modalità:
  SnapLock Compliance - Anche l'amministratore dello storage NON PUO' cancellare i dati prima
                        della scadenza - Uso: Conformità normativa stringente (SEC 17a-4, FINRA,
                        HIPAA)
  SnapLock Enterprise - L'amministratore PUO' cancellare i dati con privilegi speciali per casi
                        legittimi - Uso: Protezione ransomware, governance interna

SnapLock per la protezione dal ransomware:
  - Gli snapshot su volumi SnapLock non possono essere cancellati dal ransomware (anche se
    ottiene credenziali di amministratore)
  - Fornisce un punto di recovery inviolabile che il ransomware non può raggiungere

MULTI-ADMIN VERIFICATION (MAV)
Multi-Admin Verification richiede che le operazioni ONTAP sensibili (come la cancellazione di
snapshot, la disabilitazione di SnapLock o la modifica delle policy di protezione) debbano essere
approvate da più amministratori prima dell'esecuzione.

Come funziona:
  1. L'amministratore A richiede di cancellare uno snapshot
  2. ONTAP invia una richiesta di approvazione all'amministratore B (e opzionalmente C)
  3. L'operazione procede solo quando viene ricevuto il numero richiesto di approvazioni
  4. Tutte le richieste di approvazione e le decisioni vengono registrate nel log

Perché è importante: anche se un attaccante ransomware o un insider malintenzionato ottiene le
credenziali di un amministratore storage, non può unilateralmente cancellare i backup o disabilitare
la protezione — un secondo amministratore deve approvare.


4.4 Rafforzare la resilienza al Ransomware con le capacità ONTAP
-----------------------------------------------------------------
Il ransomware è una delle minacce più serie per l'IT enterprise. ONTAP fornisce una difesa multi-
livello contro gli attacchi ransomware.

La catena di attacco ransomware:
  1. Accesso iniziale       -> Phishing, credenziali compromesse, vulnerabilità non patchate
  2. Movimento laterale     -> L'attaccante si muove attraverso la rete verso i sistemi storage
  3. Escalation privilegi   -> L'attaccante ottiene accesso a livello admin
  4. Esfiltrazione dati     -> L'attaccante copia i dati in una posizione esterna (per doppia estorsione)
  5. Cifratura              -> L'attaccante cifra i dati e cancella i backup
  6. Richiesta di riscatto  -> Pagamento richiesto in cambio della chiave di decifratura

Le protezioni di ONTAP affrontano le fasi da 3 a 6 — assumendo che l'attaccante abbia già raggiunto
il sistema storage.

ONTAP AUTONOMOUS RANSOMWARE PROTECTION (ARP)
ARP è il motore di rilevamento ransomware AI/ML integrato in ONTAP, introdotto in ONTAP 9.10.1.

Come funziona ARP:
  1. Fase di apprendimento (30 giorni): ARP monitora i pattern normali del workload per ogni
     volume — tipi di file scritti, pattern I/O, livelli di entropia
  2. Monitoraggio attivo: ARP analizza continuamente gli I/O in arrivo rispetto alla baseline
     appresa
  3. Rilevamento anomalie: ARP segnala attività sospette, come:
     - Comparsa improvvisa di nuove estensioni di file sconosciute (file in fase di cifratura)
     - Aumento anomalo delle operazioni di rinomina file
     - Scritture di dati ad alta entropia (i dati cifrati hanno entropia più alta dei dati normali)
     - Pattern di accesso insoliti (accesso da utenti o orari inaspettati)
  4. Snapshot automatico: al rilevamento di un'anomalia, ARP crea automaticamente uno snapshot
     tamper-proof del volume interessato
  5. Alert: gli amministratori vengono notificati via email, SNMP o syslog

Capacità chiave di ARP:
  Zero-day detection       - Rileva varianti di ransomware precedentemente sconosciute in base
                             ai pattern comportamentali, non alle firme
  Preservazione prove auto - Snapshot creato immediatamente al rilevamento — prima che la
                             cifratura si diffonda ulteriormente
  Gestione falsi positivi  - Gli amministratori possono contrassegnare gli eventi come falsi
                             positivi per affinare il modello
  ARP/AI (ONTAP 9.15.1+)  - Modello AI migliorato con maggiore accuratezza e rilevamento più
                             rapido

ARP NON sostituisce gli strumenti di sicurezza (firewall, EDR, IAM) — è una ultima linea di difesa a
livello di storage che crea un punto di recovery anche quando tutte le altre difese sono state
aggirate.

SNAPSHOT IMMUTABILI
Anche senza ARP, gli snapshot standard di ONTAP forniscono una protezione critica dal ransomware:
  - Gli snapshot nella directory .snapshot sono nascosti dall'accesso CIFS/NFS standard — il
    ransomware in esecuzione in user space tipicamente non riesce a trovarli o cancellarli
  - SnapLock for SnapVault: protegge gli snapshot di backup con retention WORM — anche se il
    ransomware ottiene credenziali admin, non può cancellare gli snapshot protetti da SnapLock
    prima della scadenza
  - Tamper-proof snapshots (ONTAP 9.12.1+): una nuova funzionalità di blocco snapshot che
    contrassegna i singoli snapshot come immutabili con un periodo di retention

CIFRATURA DEI DATI A RIPOSO E IN TRANSITO

NetApp Volume Encryption (NVE):
  - Cifratura software-based dei singoli volumi usando AES-256
  - Chiavi di cifratura gestite dall'ONTAP Key Manager (OKM) o da key manager esterni (server
    KMIP come Thales, Entrust)
  - I dati vengono cifrati prima di essere scritti su disco — protegge contro il furto fisico
    dei drive

NetApp Aggregate Encryption (NAE):
  - Cifra a livello di aggregate (tutti i volumi nell'aggregate condividono una chiave)
  - Abilita la deduplication cross-volume mantenendo la cifratura
  - Più storage-efficient di NVE per ambienti con alta dedup

Self-Encrypting Drive (SED):
  - Cifratura hardware integrata nel drive
  - Complementare a NVE/NAE — fornisce un ulteriore strato di protezione

Cifratura in transito:
  - Tutto il traffico di gestione ONTAP (REST API, CLI via SSH, System Manager via HTTPS) è
    cifrato
  - Cifratura SMB 3.0 per i dati file in transito
  - IPsec per la cifratura dei dati iSCSI
  - TLS per il Cluster Interconnect ONTAP (opzionale)

ZERO TRUST ARCHITECTURE CON ONTAP
ONTAP supporta l'implementazione dei principi Zero Trust per lo storage.

"Never Trust, Always Verify" a livello di storage:
  - Multi-factor authentication (MFA) per gli account amministratore ONTAP
  - RBAC (Role-Based Access Control): ruoli granulari limitano ciò che ogni amministratore o
    service account può fare
  - Audit logging: ogni azione amministrativa registrata in un audit log immutabile
    (configurabile per l'inoltro a syslog/SIEM)
  - Autonomous Ransomware Protection: monitoraggio comportamentale continuo
  - Multi-Admin Verification: previene che la compromissione di un singolo amministratore
    causi danni irreversibili
  - Comunicazioni cifrate: tutti i percorsi di gestione e dati cifrati

ONTAP SECURITY HARDENING
ONTAP fornisce una Security Hardening Guide con best practice di configurazione:
  - Disabilitare protocolli e servizi non utilizzati
  - Applicare la versione minima TLS (TLS 1.2+)
  - Abilitare login banner e timeout di sessione
  - Configurare SNMP v3 (cifrato, autenticato) invece di SNMPv1/v2
  - Usare reti di gestione dedicate (separate dalle reti dati)
  - Abilitare la modalità FIPS 140-2 per le operazioni crittografiche
  - Integrare con Active Directory e LDAP per l'autenticazione centralizzata
