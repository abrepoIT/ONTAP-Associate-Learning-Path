ONTAP DATA MANAGEMENT FUNDAMENTALS

MODULE 1 - GESTIONE DELL'INFRASTRUTTURA STORAGE
=================================================

1.1 ONTAP System Manager
-------------------------
ONTAP System Manager è la principale interfaccia di gestione grafica web-based per i cluster ONTAP,
disponibile senza costi aggiuntivi con ogni deployment ONTAP.

Accesso: https://<cluster-management-IP-or-hostname>

System Manager si è evoluto significativamente a partire da ONTAP 9.7, passando da un'interfaccia
multi-pagina tradizionale a una moderna applicazione single-page (SPA) con workflow guidati,
raccomandazioni AI-driven e analytics integrate.

IL DASHBOARD
Il Dashboard di System Manager è la prima schermata visualizzata al login. Fornisce un riepilogo
della salute dell'intero cluster a colpo d'occhio.

Pannelli del Dashboard:
  Capacity     - Capacità raw totale, usata, disponibile, risparmi da efficienza
                 (dedup/compressione/compaction)
  Performance  - IOPS, throughput (MB/s) e latenza a livello cluster — real-time e trending
  Health       - Riepilogo salute cluster: nodi, aggregate, volumi, dischi, LIF, HA pair
  Protection   - Stato relazioni SnapMirror (healthy, lagging, broken), riepilogo Snapshot
  Network      - Stato LIF (up/down), broadcast domain, utilizzo porte
  Nodes        - CPU, memoria e utilizzo dischi per nodo; stato HA takeover
  Active Alerts - Alert EMS correnti, notifiche rischi da Active IQ
  Storage Savings - Rapporto di efficienza real-time; spazio totale risparmiato

Comportamenti chiave del Dashboard:
  - I pannelli sono interattivi — clicca qualsiasi metrica per approfondire
  - Le raccomandazioni AI-driven appaiono direttamente sul dashboard (es. "Abilita
    deduplication su vol_data per risparmiare 2,3 TB")
  - Il dashboard si aggiorna automaticamente ogni 60 secondi
  - Le notifiche di rischio Active IQ appaiono come badge colorati

ATTIVITA' PRINCIPALI DI CONFIGURAZIONE

Cluster Initial Setup:
  - Imposta nome cluster, nomi nodi e configurazione DNS
  - Configura la LIF di gestione cluster (IP, netmask, gateway)
  - Configura server NTP per la sincronizzazione dell'orario
  - Configura AutoSupport (email, consegna HTTPS)
  - Applica le license key (o ONTAP One)

Creazione Aggregate e Storage Pool:
  System Manager: Storage --> Tiers --> Add Local Tier
  - Seleziona i drive da includere
  - Scegli il tipo RAID (RAID-DP raccomandato; RAID-TEC per drive grandi)
  - ONTAP suggerisce automaticamente la dimensione ottimale del RAID group

Creazione SVM (Storage Virtual Machine):
  System Manager: Storage --> Storage VMs --> Add
  - Definisci nome SVM
  - Seleziona protocolli da abilitare (NFS, SMB, iSCSI, FC, NVMe, S3)
  - Configura root volume e LIF di rete per ogni protocollo
  - Configura name service (Active Directory, LDAP, NIS, DNS)

Volume Provisioning:
  System Manager: Storage --> Volumes --> Add
  - Seleziona SVM host e aggregate
  - Definisci nome volume, dimensione e garanzia spazio (thin o thick)
  - Imposta Snapshot policy
  - Configura export policy (NFS) o share (SMB)
  - Abilita storage efficiency (dedup, compressione, compaction)
  - Imposta policy QoS

Configurazione LUN e iGroup (SAN):
  System Manager: Storage --> LUNs --> Add
  - Crea LUN (dimensione, tipo OS, prenotazione spazio)
  - Crea o seleziona un iGroup (raccolta di initiator host)
  - Mappa LUN all'iGroup

Configurazione di Rete:
  System Manager: Network --> Overview
  - Crea e gestisci LIF (indirizzi IP per NAS/gestione; WWPN per FC)
  - Configura VLAN e interface group (LACP)
  - Gestisci broadcast domain e routing

MONITORAGGIO DELL'INFRASTRUTTURA STORAGE

Monitoraggio della Capacità:
  - Vista capacità volumi: percentuale usata, spazio disponibile, spazio snapshot,
    overhead thin provisioning
  - Capacità aggregate: usata/disponibile con dettaglio per volume
  - Efficienza: spazio totale risparmiato e rapporto di efficienza per volume e aggregate
  - Previsione capacità: proiezione basata su trend dei giorni alla saturazione

Monitoraggio delle Performance:
  - Grafici IOPS/throughput/latenza real-time per: cluster, nodo, aggregate, volume, LUN, LIF
  - Dettaglio latenza: lettura vs scrittura
  - Top workload: identifica volumi/LUN che consumano più risorse
  - Utilizzo QoS: quanto del ceiling QoS viene consumato

Monitoraggio della Protezione:
  System Manager: Protection --> Relationships
  - Visualizza tutte le relazioni SnapMirror con stato (healthy, lagging, idle, broken-off)
  - Lag time per relazione (quanto è indietro la destinazione)
  - Ultimo trasferimento riuscito e dimensione trasferimento
  - Operazioni con un click: initialize, update, quiesce, break, resync

Monitoraggio della Salute:
  System Manager: Cluster --> Overview --> Alerts
  - Alert EMS attivi con severità, descrizione e azione raccomandata
  - Salute nodi (uptime, CPU, memoria, errori disco)
  - Salute dischi (dischi guasti, spare disponibili, stato ricostruzione RAID)
  - Stato HA pair (normale, takeover in corso, in attesa di giveback)

PROTEZIONE DELL'INFRASTRUTTURA STORAGE

Gestione Snapshot:
  System Manager: Storage --> Volumes --> [seleziona volume] --> Protection
  - Visualizza snapshot esistenti (nome, tempo creazione, dimensione)
  - Crea snapshot manuali, imposta o modifica Snapshot policy
  - Ripristina da snapshot (singolo file o volume revert completo)

Configurazione SnapMirror e SnapVault:
  System Manager: Protection --> Relationships --> Add
  - Crea relazioni SnapMirror/SnapVault (richiede cluster in peer)
  - Seleziona tipo relazione (Mirror, Vault, Mirror and Vault), policy e schedule

Protezione Ransomware (ARP):
  System Manager: Storage --> Volumes --> [volume] --> Security
  - Abilita/disabilita Autonomous Ransomware Protection per volume
  - Visualizza stato ARP (learning, active)
  - Esamina alert ARP ed eventi di attacco sospetti

Cifratura:
  System Manager: Security --> Encryption
  - Abilita NetApp Volume Encryption (NVE) per volume
  - Configura key management server (OKM o KMIP esterno)
  - Verifica stato cifratura di tutti i volumi


1.2 ONTAP CLI
--------------

PERCHE' LA CLI E' ESSENZIALE
Sebbene System Manager gestisca la grande maggioranza delle operazioni quotidiane, la CLI
ONTAP rimane essenziale per:

  Automazione e scripting    - I comandi CLI possono essere scriptati in bash, Python o Ansible
  Diagnostica avanzata       - Alcuni comandi diagnostici sono disponibili solo ai livelli
                               advanced o diagnostic
  Operazioni bulk            - Gestire centinaia di volumi o LUN è più veloce via scripting CLI
  Accesso remoto             - SSH funziona senza browser; utile in ambienti chiusi
  Documentazione             - I comandi CLI possono essere documentati nei runbook
  Correlazione API           - Ogni comando CLI mappa 1:1 su una chiamata REST API
  Filtri complessi           - I comandi show supportano filtri ricchi (-fields, grep, -state)

Metodi di accesso CLI:
  SSH             - ssh admin@<cluster-mgmt-IP>            - Accesso remoto standard
  Console seriale - Cavo seriale alla porta console del nodo - Accesso di emergenza
  SP/BMC console  - system node run-console                - Accesso out-of-band
  ONTAP Shell (SP) - system service-processor ssh          - Diagnostica a basso livello

OPERAZIONI SPECIALIZZATE VIA CLI

Operazioni bulk sui volumi (creazione di 20 volumi in un loop — non fattibile in GUI):
  for i in $(seq 1 20); do
    volume create -vserver svm_prod -volume vol_app_${i} -aggregate aggr1
      -size 500GB -junction-path /app_data/${i} -snapshot-policy default
      -space-guarantee none
  done

Diagnostica avanzata:
  set -privilege advanced
  wafl scan status
  wafl iron status
  statistics show -object disk -counter disk_busy
  vserver audit enable -vserver svm1

Diagnostica di rete:
  network ping -lif data_lif1 -destination 192.168.1.100 -count 5
  network traceroute -lif cluster_mgmt -destination 8.8.8.8
  statistics show -object lif -instance data_lif1 -counter bytes_recv,bytes_sent

Analisi dell'efficienza storage:
  volume efficiency show -fields policy,state,dedupe-space-saved,compression-space-saved
  volume efficiency start -vserver svm1 -volume vol_data -scan-old-data true
  volume show-footprint -vserver svm1 -volume vol_data


1.3 Active IQ Unified Manager
-------------------------------
Active IQ Unified Manager (AIQUM) è un'applicazione di gestione e monitoraggio dedicata per gli
ambienti ONTAP. A differenza di System Manager (che gestisce un singolo cluster), AIQUM fornisce una
vista centralizzata su più cluster ONTAP da un'unica interfaccia.

Deployment: VM Linux, Windows Server, o appliance VMware OVA — deployata on-premises
Accesso: https://<unified-manager-IP>

DASHBOARD SUMMARY DEL DATA CENTER STORAGE
Pannelli del Dashboard di gestione:
  Capacity       - Capacità aggregata su tutti i cluster gestiti; volumi e aggregate quasi
                   pieni
  Performance    - Riepilogo performance cross-cluster; top-N workload per IOPS, latenza,
                   throughput
  Protection     - Salute relazioni SnapMirror e SnapVault su tutti i cluster
  Security       - Stato conformità sicurezza (cifratura, account admin, certificati SSL)
  Workloads      - Workload attivi con confronto performance vs soglia policy
  Events         - Incident ed eventi attivi su tutti i sistemi gestiti

Differenziatori chiave AIQUM vs System Manager:
  - Gestisce più cluster simultaneamente (System Manager = un cluster alla volta)
  - Dati storici di performance fino a 13 mesi
  - Alerting basato su soglie: definisci soglie personalizzate di performance e capacità
  - Gestione eventi: crea, assegna, riconosci e risolvi incident
  - Capacity planning: forecasting basato su trend su tutto il data center

PROVISIONING STORAGE VIA AIQUM

Performance Service Level (PSL):
I PSL definiscono le caratteristiche di performance storage (IOPS/TB, latenza attesa):
  Extreme     - IOPS/TB: 12.500+   - Latenza: <1 ms  - Uso: Database Tier-1, VDI
  Performance - IOPS/TB: 4.000     - Latenza: <2 ms  - Uso: Applicazioni produzione
  Value       - IOPS/TB: 128       - Latenza: <17 ms - Uso: File share, archiviazione

Quando si effettua il provisioning via AIQUM, l'amministratore seleziona un PSL — AIQUM seleziona
automaticamente l'aggregate appropriato e configura QoS per applicare l'SLA.

AZIONI DI GESTIONE: FIX-IT REMEDIATION
Una delle funzionalità più potenti di AIQUM è la remediation automatica — la capacità non solo di
rilevare i problemi ma di risolverli direttamente dall'interfaccia di gestione.

Come funziona Fix-It:
  1. AIQUM rileva una condizione che si discosta dalle best practice o supera una soglia
  2. Viene generato un evento con severità (critical, error, warning, information)
  3. La pagina dei dettagli dell'evento mostra: causa root, impatto e azione Fix-It suggerita
  4. L'amministratore esamina la raccomandazione e clicca Fix It
  5. AIQUM esegue la modifica di configurazione ONTAP via REST API
  6. L'evento viene risolto e registrato nel log

Esempi di remediation Fix-It:
  Volume quasi pieno           - Aumenta dimensione volume o abilita autogrow
  Snapshot oltre la riserva    - Elimina snapshot più vecchi o riduci retention
  Lag relazione SnapMirror     - Attiva un aggiornamento SnapMirror immediato
  Deduplication volume disab.  - Abilita deduplication inline
  Share CIFS senza controllo   - Applica permessi share raccomandati
  Export policy NFS troppo ampia - Restringe le regole della export policy

2.1 Moduli NetApp per Ansible
-------------------------------
Ansible è un framework di automazione IT open-source che usa playbook YAML human-readable per
definire stati di sistema desiderati. NetApp fornisce una collection Ansible certificata e ufficiale
per ONTAP (netapp.ontap) disponibile su Ansible Galaxy.

COSA SI PUO' AUTOMATIZZARE CON ANSIBLE?
Praticamente ogni operazione di gestione ONTAP può essere automatizzata:

  Gestione cluster      - Configura NTP, DNS, SNMP, AutoSupport, licenze
  Gestione aggregate    - Crea/elimina aggregate, aggiungi drive, monitora capacità
  Gestione SVM          - Crea/configura/elimina SVM, abilita protocolli, unisci AD
  Gestione volumi       - Crea/elimina/ridimensiona volumi, imposta efficienza, QoS
  Gestione LUN          - Crea/elimina/ridimensiona LUN, crea igroup, mappa LUN
  NFS/SMB               - Crea export, share, configura security style, ACL
  SnapMirror            - Crea/inizializza/aggiorna/interrompi/resync relazioni
  Snapshot              - Crea/elimina snapshot, gestisci policy
  Utenti e RBAC         - Crea utenti, ruoli, applica requisiti MFA
  Networking            - Crea LIF, VLAN, ifgroup, broadcast domain
  Sicurezza             - Abilita NVE, configura key manager, abilita ARP
  FabricPool            - Configura object store, imposta policy di tiering

Installazione:
  ansible-galaxy collection install netapp.ontap

Componenti core che lavorano insieme:
  Playbook (.yml):      Definiscono COSA deve essere configurato; usano variabili e chiamano
                        ruoli e task
  Ruoli:                Bundle riutilizzabili di task (es. "ontap_nas_setup")
  NetApp ONTAP Collection: 200+ moduli (na_ontap_volume, na_ontap_lun, ecc.); ogni modulo
                        mappa chiamate ONTAP REST API
  Ansible Vault:        Storage cifrato per password e segreti
  Inventory:            Definisce cluster ONTAP target e parametri di connessione

Esempio di playbook — crea un volume NFS con efficienza e snapshot policy:
  - name: Create SVM
    na_ontap_svm: state=present name=svm_app root_volume_aggregate=aggr1 ...

  - name: Create volume
    na_ontap_volume: state=present name=vol_app_data vserver=svm_app
      aggregate_name=aggr1 size=2000 size_unit=gb space_guarantee=none
      snapshot_policy=default junction_path=/app_data efficiency_policy=auto ...

  - name: Create NFS export policy rule
    na_ontap_export_policy_rule: state=present name=default vserver=svm_app
      client_match=10.10.0.0/24 ro_rule=sys rw_rule=sys super_user_security=sys ...


2.2 ONTAP REST API
-------------------
La REST API di ONTAP fornisce accesso programmatico a tutte le funzioni di gestione ONTAP tramite
metodi HTTP standard (GET, POST, PATCH, DELETE) usando payload JSON.

Perché REST API?
  - Language-agnostic: qualsiasi linguaggio con supporto HTTP (Python, Go, JavaScript,
    PowerShell, curl)
  - Integrazione con qualsiasi piattaforma di automazione (Ansible, Terraform, Jenkins,
    GitHub Actions)
  - Funzionalità identica alla CLI — tutto ciò che si può fare in CLI, si può fare via API
  - Supporta esecuzione job sincrona e asincrona

URL base API: https://<cluster-mgmt-IP>/api
Documentazione integrata: https://<cluster-mgmt-IP>/docs/api (Swagger/OpenAPI UI)

Categorie API ONTAP:
  cluster      - /api/cluster             - Info cluster, nodi, licenze, job
  storage      - /api/storage/*           - Volumi, aggregate, LUN, dischi, qtree
  svm          - /api/svm/svms            - Operazioni CRUD sugli SVM
  network      - /api/network/*           - LIF, porte, route, IPspace
  protocols    - /api/protocols/*         - Export NFS, share CIFS, iSCSI, FC, S3
  snapmirror   - /api/snapmirror/*        - Relazioni, policy, trasferimenti
  security     - /api/security/*          - Utenti, ruoli, cifratura, certificati
  support      - /api/support/*           - AutoSupport, EMS, SNMP
  application  - /api/application/*       - Template applicazioni (SAP HANA, Oracle, ecc.)

Operazioni asincrone: le operazioni lunghe (spostamento volumi, inizializzazione SnapMirror)
restituiscono un oggetto job che può essere monitorato tramite polling sull'endpoint job URL.


2.3 Active IQ Unified Manager REST API
----------------------------------------
AIQUM espone anche una REST API che fornisce accesso programmatico alle sue capacità di gestione,
monitoraggio e provisioning.

Categorie API AIQUM:
  datacenter        - Accede ai dati di monitoraggio (cluster, nodi, volumi, aggregate, LIF)
  storage-provider  - Provisioning storage (workload, PSL, storage efficiency policy)
  administration    - Gestisce utenti AIQUM, datasource, soglie
  gateway           - Chiamate pass-through direttamente all'API REST ONTAP tramite proxy AIQUM

INTEGRAZIONE SERVICENOW
AIQUM si integra nativamente con ServiceNow per workflow ITSM enterprise:
  - Gli eventi AIQUM vengono automaticamente convertiti in incident ServiceNow
  - La severità dell'incident mappa alla severità dell'evento AIQUM (critical=P1, error=P2,
    warning=P3)
  - I tecnici ServiceNow possono visualizzare il contesto storage direttamente nell'incident
  - La risoluzione dell'incident in ServiceNow può attivare remediation Fix-It di AIQUM
  - Le change request in ServiceNow possono attivare workflow di provisioning storage AIQUM

3.1 SnapCenter
---------------
SnapCenter è la piattaforma di backup e recovery enterprise application-aware di NetApp, che integra
le capacità native Snapshot, SnapMirror e SnapVault di ONTAP con la consistenza a livello
applicativo.

Interfacce di gestione SnapCenter:
  SnapCenter Web UI     - Interfaccia grafica principale (browser web HTTPS)
  SnapCenter REST API   - Accesso programmatico a tutte le operazioni
  SnapCenter CLI        - Interfaccia a riga di comando per scripting
  PowerShell cmdlet     - Integrazione Windows PowerShell

Accesso: https://<snapcenter-server>:8146

IL DASHBOARD DI SNAPCENTER
Componenti del Dashboard:
  Summary tile     - Backup totali, backup falliti, restore riusciti, clone attivi
  Activity pane    - Job di backup/restore/clone recenti e in esecuzione
  Topology view    - Rappresentazione visuale di volumi sorgente, snapshot, relazioni
                     SnapMirror e copie vault
  Protection status - Quali risorse sono protette vs non protette
  Capacity         - Storage consumato da snapshot e clone gestiti da SnapCenter
  Events e alerts  - Guasti recenti, warning ed eventi informativi

La topology view è particolarmente potente — mostra la catena completa di data protection per
una risorsa selezionata:
  Database Host
    +-- File database (su volume ONTAP)
        +-- Snapshot 1 (hourly.2025-09-01_1200)
        +-- Snapshot 2 (daily.2025-09-01_2200)
            +-- SnapMirror --> Copia DR (cluster secondario)
                +-- SnapVault --> Archivio a lungo termine (vault cluster)

SNAPCENTER: ROLE-BASED ACCESS CONTROL (RBAC)
SnapCenter implementa un modello RBAC granulare:

Ruoli SnapCenter integrati:
  SnapCenter Admin           - Accesso completo a tutte le risorse e impostazioni
  App Backup and Clone Admin - Operazioni di backup e clone; no modifiche infrastruttura
  Backup and Clone Viewer    - Read-only: visualizza backup, job, report
  Infrastructure Admin       - Gestisce infrastruttura SnapCenter (host, credenziali, storage)

I ruoli personalizzati possono essere creati per limitare precisamente i permessi a specifiche
risorse, operazioni o sistemi storage.

Integrazione con Active Directory:
  - Gli utenti SnapCenter possono essere utenti AD individuali o gruppi AD
  - I permessi si applicano a tutti i membri del gruppo AD
  - Elimina la necessità di gestire account SnapCenter separatamente da AD

PERCHE' SNAPCENTER PER LA DATA PROTECTION?
  Backup app-consistent richiedono scripting  --> Plugin integrati gestiscono quiescing
                                                  automatico per Oracle, SQL, VMware, HANA
  Tool separati per ogni applicazione         --> Unica piattaforma gestisce tutto
  Admin storage come collo di bottiglia       --> RBAC abilita self-service restore
  Nessuna visibilità sul catalogo backup      --> Catalogo centralizzato e ricercabile
  Gestione manuale SnapMirror per DR          --> Automazione policy snapshot->mirror->vault
  Verifica backup richiede test manuali       --> Verifica automatizzata basata su clone
  No integrazione con ticketing               --> REST API abilita integrazione ServiceNow


3.2 ONTAP Tools for VMware vSphere
------------------------------------
ONTAP tools for VMware vSphere (OTV) è un plug-in per vCenter Server che fornisce integrazione
profonda tra lo storage ONTAP e gli ambienti VMware vSphere.

PERCHE' ONTAP TOOLS FOR VMWARE VSPHERE?
Senza OTV, gli amministratori VMware devono:
  - Richiedere manualmente agli admin storage il provisioning dei datastore
  - Non avere visibilità sullo storage ONTAP che supporta le loro VM
  - Non poter applicare policy QoS storage da vCenter
  - Non poter automatizzare il failover DR per le VM

Con OTV:
  - Gli admin VMware possono fare provisioning dei datastore direttamente da vCenter
  - La topologia storage per ogni VM è visibile in vCenter
  - Policy QoS storage applicate via VMware SPBM
  - Failover DR per le VM automatizzato tramite integrazione VMware SRM

DYNAMIC DATA MANAGEMENT PER LE VM

VASA (vSphere APIs for Storage Awareness) Provider:
OTV agisce come VASA Provider, esponendo le capacità storage ONTAP a vSphere:
  - Definisce storage capability (es. "Flash - alta IOPS - cifrato") come profili storage
    vSphere
  - Gli admin VMware assegnano profili storage alle VM (SPBM)
  - vSphere posiziona automaticamente le VM su storage ONTAP che corrisponde al profilo
  - Controllo di conformità continuo: alert se lo storage di una VM non soddisfa più la policy

Virtual Volume (vVol):
  - Ogni disco VM diventa un oggetto ONTAP individuale (LUN o file NFS)
  - Gestione storage granulare per VM (snapshot, QoS, cifratura per VMDK)
  - Elimina la gestione a livello LUN; lo storage segue la VM

IL DASHBOARD DI OTV
  Datastores    - Tutti i datastore supportati da ONTAP, utilizzo, stato conformità
  VM            - Consumo storage per VM, latenza, IOPS
  Storage System - Cluster ONTAP connessi e la loro salute
  vVols         - Inventario Virtual Volume e conformità policy
  Alarms        - Allarmi storage visibili in vCenter

DR CON SRA E VASA PROVIDER

SRA (Site Recovery Adapter):
OTV include un SRA che si integra con VMware Site Recovery Manager (SRM):

  Sito A (Protetto)                   Sito B (Recovery)
  vCenter A + SRM A                   vCenter B + SRM B
  Cluster ONTAP A                     Cluster ONTAP B
  VM su volumi ONTAP                  Copie DR (SnapMirror)
       |                                     |
       +---------- SnapMirror Async -------->+
       +--- SRA comunica con entrambi i sistemi ONTAP tramite OTV REST API

Workflow DR SRM:
  1. SRM rileva guasto del sito primario
  2. SRM chiama SRA per interrompere SnapMirror sul secondario
  3. SRA rimonta i volumi sul Sito B
  4. SRM accende le VM al Sito B nell'ordine corretto (piano di recovery)
  5. Mappature di rete applicate (riconfigurazione IP se necessario)

Test failover SRM:
  SRM può testare i piani DR in modo non distruttivo usando FlexClone (ONTAP crea un clone del
  volume DR — nessun impatto sulla relazione SnapMirror di produzione). Dopo il test, il clone
  viene eliminato.

4.1 Data Infrastructure Insights (ex Cloud Insights)
------------------------------------------------------
NetApp Data Infrastructure Insights (DII) è una piattaforma di monitoraggio e analytics
infrastrutturale cloud-based che fornisce osservabilità full-stack su storage, compute e rete
— coprendo sia infrastruttura NetApp che di terze parti.

Monitoraggio tradizionale vs Data Infrastructure Insights:
  Tool per vendor separati      --> Unica piattaforma per tutta l'infrastruttura
  No correlazione tra layer      --> Correla VM -> datastore -> volume -> latenza disco
  Reattivo (alert dopo guasto)  --> Predittivo (identifica trend prima del guasto)
  Capacity planning manuale     --> Forecasting AI-driven e rilevamento anomalie
  Visibilità limitata sul cloud  --> On-premises + cloud in un'unica vista

I TRE MODULI DI DATA INFRASTRUCTURE INSIGHTS

1. Modulo Observability:
  - Topologia full-stack: scopre e mappa automaticamente le relazioni tra host, VM, datastore,
    volumi, aggregate e dischi

MODULE 5 - SOLUZIONI NETAPP DI CYBER-RESILIENCE
=================================================

5.1 Cyber Resilience
---------------------

LA PROTEZIONE CYBER-RESILIENCE
La cyber resilience è la capacità di un'organizzazione di anticipare, resistere, recuperare e
adattarsi a condizioni, attacchi o compromissioni informatiche avverse. Va oltre la cybersecurity
tradizionale (prevenire gli attacchi) per includere la capacità di recuperare rapidamente e
completamente quando la prevenzione fallisce.

Il cambiamento di mentalità:
  Sicurezza tradizionale: "Possiamo prevenire tutti gli attacchi"
  Cyber Resilience:       "Le violazioni sono inevitabili; dobbiamo recuperare velocemente"

  Focus tradizionale:  difesa del perimetro
  Focus cyber resilience: perimetro + difesa a livello dati + recovery

  Metrica tradizionale: numero di attacchi bloccati
  Metrica cyber resilience: tempo per recuperare, perdita dati evitata

  Backup tradizionale: è un ripensamento
  Backup cyber resilience: backup e immutabilità sono difese in prima linea

Data-Centric Cyber Resilience:
NetApp posiziona lo storage come ultima linea di difesa — anche se tutti gli altri layer di
sicurezza vengono aggirati, il layer storage fornisce:
  1. Punti di recovery immutabili (snapshot che gli attaccanti non possono eliminare)
  2. Rilevamento anomalie comportamentali (ARP rileva cifratura in corso)
  3. Copie di backup air-gapped (backup cloud isolati dall'on-premises)
  4. Governance e compliance (storage WORM per requisiti normativi)


5.2 Protezione Ransomware
--------------------------

LO STORAGE E' L'ULTIMA LINEA DI DIFESA
La catena di attacco ransomware moderna:
  Stage 1: Accesso iniziale       --> Email phishing, VPN compromessa, CVE non patchato
  Stage 2: Persistenza            --> Backdoor, account admin non autorizzati
  Stage 3: Ricognizione           --> Mappa rete interna, identifica sistemi backup
  Stage 4: Movimento laterale     --> Si avvicina ai sistemi storage e backup
  Stage 5: Escalation privilegi   --> Ottiene credenziali admin storage
  Stage 6: Distruzione backup     --> Elimina backup, snapshot, repliche SnapMirror
                                      <-- LO STORAGE DEVE RESISTERE QUI
  Stage 7: Esfiltrazione dati     --> Copia dati sensibili su server esterno (doppia estorsione)
  Stage 8: Cifratura              --> Cifra tutti i dati accessibili
                                      <-- LO STORAGE RILEVA QUI (ARP)
  Stage 9: Richiesta riscatto     --> Pagamento richiesto per la chiave di decifratura

Difese storage agli stage 6 e 8:
  Stage 6: SnapLock e Multi-Admin Verification prevengono la cancellazione dei backup
  Stage 8: ARP rileva la cifratura e crea uno snapshot immutabile

Capacità di protezione ransomware NetApp:
  Detection            - ARP: rilevamento comportamentale AI; snapshot automatico su anomalia
  Backup immutabili    - SnapLock Enterprise/Compliance: snapshot e vault WORM
  Protezione dual-admin - MAV: le operazioni critiche richiedono 2 approvazioni admin
  Copie air-gapped     - BlueXP Backup con DataLock: copie cloud isolate dall'on-premises
  Isolamento rete      - Controlli rete a livello SVM: SVM compromessa non raggiunge le altre
  Controllo accessi    - RBAC + MFA: limita il raggio di esplosione delle credenziali compromesse
  Audit trail          - FPolicy + audit logging: tutti gli accessi file registrati
  CIFS sicuro          - SMB signing e cifratura: previene attacchi man-in-the-middle

APPROCCIO IN 5 PASSI ALLA PROTEZIONE RANSOMWARE

Passo 1 - IDENTIFY (Identificare):
  - Inventaria tutti gli asset dati e classificali per sensibilità
  - Mappa i flussi di dati e i pattern di accesso
  - Usa BlueXP Classification per identificare automaticamente dati sensibili (PII, finanziari)
  - Comprendi quali utenti e servizi accedono a quali dati

Passo 2 - PROTECT (Proteggere):
  - Abilita ARP su tutti i volumi di produzione
  - Abilita Multi-Admin Verification per le operazioni critiche
  - Applica MFA per tutti gli account amministratore storage
  - Applica SnapLock Enterprise alle copie vault di backup
  - Configura ONTAP security hardening (disabilita protocolli inutilizzati, TLS 1.2+, SNMP v3)
  - Integra con SIEM (Splunk, QRadar) per monitoraggio sicurezza centralizzato

Passo 3 - DETECT (Rilevare):
  - ARP monitora le anomalie comportamentali (nuove estensioni file, scritture ad alta entropia,
    rinomina massiva)
  - FPolicy monitora e registra tutti gli eventi di accesso file NAS
  - Data Infrastructure Insights Workload Security monitora i pattern di accesso utente
  - Active IQ genera alert di rischio sicurezza per volumi non cifrati, certificati scadenti

Passo 4 - RESPOND (Rispondere):
  - ARP genera un alert EMS e crea uno snapshot immutabile al rilevamento di anomalie
  - L'amministratore riceve notifica (email/SNMP/syslog/ServiceNow)
  - Isola: disabilita account utenti compromessi tramite AD; blocca utenti specifici via DII
  - Preserva prove: non eliminare gli snapshot generati da ARP; cattura i log di rete
  - Coinvolgi il supporto NetApp e il team di risposta agli incident di sicurezza
  - Valuta: determina l'ultimo snapshot known-good prima dell'inizio della cifratura

Passo 5 - RECOVER (Recuperare):
  - Identifica l'ultimo snapshot pulito (prima dell'attività ransomware)
  - Ripristina da Snapshot ONTAP (minuti), SnapVault (ore) o BlueXP Backup (cloud)
  - Se i backup on-premises sono stati compromessi: ripristina da backup cloud protetto
    da BlueXP DataLock
  - Valida l'integrità dei dati ripristinati
  - Esegui analisi causa root e patcha il vettore di attacco iniziale
  - Documenta le lezioni apprese e aggiorna il piano di risposta


5.3 Backup
-----------

BACKUP E RESTORE — REGOLA 3-2-1-1-0
Una strategia di backup completa affronta tutti i layer di data protection:

  3 copie dei dati:
    Copia 1: Dati di produzione primari (su ONTAP)
    Copia 2: Copia secondaria (SnapVault su cluster secondario o CVO)
    Copia 3: Terza copia (object storage cloud via BlueXP Backup)

  2 tipi di supporti diversi:
    Flash/disco (ONTAP primario + secondario)
    Object storage cloud (S3, Blob, GCS)

  1 copia offsite:
    Il backup cloud è intrinsecamente offsite

  1 copia air-gapped / immutabile:
    BlueXP DataLock + S3 Object Lock = immutabile, non può essere eliminata

  0 backup non verificati:
    Test di restore periodici (verifica basata su clone SnapCenter)

PERCHE' BLUEXP BACKUP AND RECOVERY?
  Richiede HW secondario on-prem       --> Nessun HW secondario; usa object storage cloud
  Agenti di backup separati per app    --> Agentless; integrazione diretta con ONTAP
  Capacity planning manuale per backup --> Scalabilità cloud illimitata
  Difficile imporre immutabilità       --> DataLock fornisce protezione livello WORM
  Alto costo per retention a L.T.      --> Tier archivio riduce costi retention dell'80%+
  Singolo punto di guasto se on-prem   --> Copia cloud offsite sempre disponibile
    è compromessa


5.4 Disaster Recovery
----------------------
NetApp fornisce uno spettro di soluzioni DR adatte a diversi requisiti RPO/RTO:

  SnapMirror Async       - RPO: Minuti  - RTO: Minuti-Ore    - DR standard tra siti
  SnapMirror Sync        - RPO: Zero    - RTO: Minuti        - DR senza perdita dati
  SnapMirror Active Sync - RPO: Zero    - RTO: Quasi zero    - Workload SAN business-critical
  MetroCluster           - RPO: Zero    - RTO: ~30 secondi   - BC/DR enterprise tutti protocolli
  Cloud Volumes ONTAP    - RPO: Minuti  - RTO: Minuti-Ore    - Target DR cloud-based
  BlueXP Disaster Recovery - RPO: Min  - RTO: Minuti        - DR workload VMware su cloud

BLUEXP DISASTER RECOVERY
BlueXP Disaster Recovery è un servizio gestito che orchestra il DR per i workload VMware verso il
cloud o tra siti on-premises usando ONTAP SnapMirror.

Come funziona:
  1. BlueXP scopre le VM VMware e il loro storage ONTAP sottostante
  2. L'amministratore crea un piano DR (quali VM, in quale ordine, mappature di rete)
  3. BlueXP monitora continuamente la salute della replica SnapMirror
  4. Su evento DR: BlueXP interrompe SnapMirror, monta volumi su vCenter DR, accende VM
  5. Dopo il recovery: BlueXP gestisce il failback al sito primario

Vantaggi rispetto al DR SnapMirror manuale:
  - Orchestrazione automatizzata (nessun comando ONTAP manuale durante la crisi)
  - Piani DR pre-testati (test failover non distruttivo usando FlexClone)
  - RTO misurato in minuti invece di ore
  - Unica interfaccia BlueXP gestisce l'intero workflow DR
