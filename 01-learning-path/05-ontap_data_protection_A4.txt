ONTAP DATA PROTECTION FUNDAMENTALS
====================================


MODULE 1 - PANORAMICA DELLA DATA PROTECTION ONTAP
===================================================

1.1 Funzionalità di Data Protection ONTAP
-------------------------------------------

PERCHE' LA DATA PROTECTION E' IMPORTANTE
Le organizzazioni affrontano un insieme sempre più ampio di sfide di data protection:

  Errore umano           - Cancellazione accidentale, configurazione errata, sovrascritture
                           involontarie - Impatto: perdita dati, violazione compliance
  Guasto hardware        - Guasto drive, controller, alimentatore
                           Impatto: downtime, potenziale perdita dati
  Corruzione software    - Bug OS, errori applicativi, corruzione filesystem
                           Impatto: inconsistenza dati, downtime
  Minacce cybersecurity  - Ransomware, malware, insider threat
                           Impatto: cifratura/perdita massiva dati, estorsione
  Disastro a livello sito - Incendio, alluvione, terremoto, interruzione elettrica
                           Impatto: perdita completa del data center
  Requisiti normativi    - GDPR, HIPAA, SOX, PCI-DSS, SEC 17a-4
                           Impatto: sanzioni, responsabilità legale

FATTORI DI PIANIFICAZIONE DELLA DATA PROTECTION
Prima di selezionare una soluzione di data protection, i team IT devono valutare:

1. Recovery Point Objective (RPO) — Quanti dati possiamo permetterci di perdere?
  RPO = 0:      zero perdita dati — ogni scrittura deve essere confermata su un sistema secondario
  RPO = minuti: la replica periodica è accettabile
  RPO = ore:    backup giornalieri o notturni possono essere sufficienti
  RPO = giorni: solo per dati a basso valore, raramente modificati

2. Recovery Time Objective (RTO) — Con quale rapidità i servizi devono essere ripristinati?
  RTO = secondi:  failover automatico senza intervento umano
  RTO = minuti:   failover manuale con ambiente DR pre-configurato
  RTO = ore:      ripristino da backup, riconfigurazione degli host
  RTO = giorni:   ricostruzione da zero (inaccettabile per sistemi mission-critical)

3. Scope del Recovery — Cosa deve essere recuperabile?
  - Singolo file o directory
  - Intero volume
  - Intero SVM (tutti i volumi + configurazione)
  - Intero sistema di storage
  - Intero sito del data center

4. Requisiti di consistenza dei dati:
  Crash-consistent:        i dati sono nello stato in cui sarebbero se la corrente fosse
                           stata interrotta bruscamente — accettabile per la maggior parte di
                           OS e database con journaling
  Application-consistent:  l'applicazione viene messa in quiesce prima dello snapshot —
                           richiesto per database (Oracle, SQL Server) e VM
  Replication-consistent:  tutti i sistemi dipendenti vengono catturati nello stesso momento

5. Requisiti di retention:
  - Quanti punti di recovery devono essere disponibili?
  - Fino a quanto indietro nel tempo si deve poter recuperare?
  - Sono richieste copie immutabili (WORM) per la compliance?
  - Sono richieste copie offsite o air-gapped?

6. Vincoli di costo e infrastruttura:
  - L'hardware secondario può essere dedicato alla protezione, o deve essere condiviso?
  - Lo storage cloud è disponibile come target di backup?
  - Qual è la banda di rete disponibile per la replica?
  - Qual è l'overhead di storage accettabile per le copie di protezione?

CATEGORIE DI SOLUZIONI DI DATA PROTECTION ONTAP
ONTAP organizza le sue capacità di data protection in tre categorie strategiche:

  BACKUP & ARCHIVE           DISASTER RECOVERY          BUSINESS CONTINUITY
  Snapshot                   SnapMirror Async            SM Active Sync
  SnapVault                  SnapMirror Sync             MetroCluster
  SnapCenter                 SVM DR
  SnapMirror Cloud           SnapMirror S3
  BlueXP Backup

  RPO: Ore-Giorni            RPO: Minuti-Zero            RPO: Zero
  RTO: Ore                   RTO: Minuti-Ore             RTO: Zero-Secondi


1.2 ONTAP Snapshot Technology
-------------------------------

COS'E' UNO SNAPSHOT?
Uno Snapshot ONTAP è un'immagine point-in-time istantanea e read-only di un volume. Gli snapshot
sono il building block fondamentale di tutta la data protection ONTAP — ogni altra funzionalità di
protezione (SnapMirror, SnapVault, SnapCenter) si basa sugli Snapshot come unità di dati trasferiti
o conservati.

COME FUNZIONANO GLI SNAPSHOT: WRITE ANYWHERE FILE LAYOUT (WAFL)
Il filesystem interno di ONTAP, WAFL, è progettato per rendere la creazione degli Snapshot
istantanea e space-efficient.

Principio chiave di WAFL: i nuovi dati non vengono mai scritti sopra quelli esistenti. Ogni
scrittura va su un blocco nuovo, mai usato in precedenza. Questo significa:
  - Creare uno Snapshot è solo un'operazione di metadata (cattura la mappa blocchi corrente)
  - Non è necessario copiare dati quando viene preso uno Snapshot
  - La creazione di uno Snapshot richiede millisecondi indipendentemente dalla dimensione del
    volume

Meccanica Copy-on-Write:

  STATO 1: Prima di qualsiasi snapshot
  Mappa blocchi filesystem attivo:  [A1][B1][C1][D1][E1]
  Snapshot:                         (nessuno)
  Blocchi fisici:                   [A1][B1][C1][D1][E1]

  STATO 2: Snapshot preso (istantaneo — zero spazio consumato)
  Mappa blocchi filesystem attivo:  [A1][B1][C1][D1][E1]  <-- stessi blocchi
  Mappa blocchi snapshot:           [A1][B1][C1][D1][E1]  <-- stessi blocchi (condivisi)
  Blocchi fisici:                   [A1][B1][C1][D1][E1]  <-- nessuna duplicazione

  STATO 3: Il filesystem attivo modifica A e C
  Mappa blocchi filesystem attivo:  [A2][B1][C2][D1][E1]  <-- nuovi blocchi A2, C2
  Mappa blocchi snapshot:           [A1][B1][C1][D1][E1]  <-- punta ancora ai vecchi blocchi
  Blocchi fisici:          [A1][B1][C1][D1][E1][A2][C2]   <-- vecchi + nuovi coesistono

  SPAZIO SNAPSHOT = dimensione di [A1] + [C1] (solo i blocchi modificati)

Efficienza dello spazio:
  - Uno Snapshot appena creato consuma zero spazio aggiuntivo
  - Lo spazio viene consumato solo man mano che i dati attivi divergono dallo stato snapshot
  - Gli snapshot di workload read-heavy rimangono molto piccoli per lungo tempo

Consumo di spazio (esempio):
  Età snapshot:  1 ora    4 ore    24 ore    1 settimana
  Spazio usato:  ~0,5%    ~2%      ~5%       ~15%
  (varia molto in base al tasso di modifica dei dati)

CAPACITA' DEGLI SNAPSHOT
  Max snapshot per volume:    1.023
  Tempo di creazione:         Millisecondi (indipendentemente dalla dimensione del volume)
  Spazio minimo consumato:    Zero (immediatamente dopo la creazione)
  Granularità:                Per-volume
  Accesso (NFS):              Via directory nascosta .snapshot alla radice del volume montato
  Accesso (SMB):              Via ~snapshot o funzionalità "Versioni precedenti" di Windows
  Accesso (SAN):              Via clone LUN o SnapRestore (nessun accesso diretto client)
  Granularità restore:        Singolo file, LUN, o intero volume

SNAPSHOT POLICY E SCHEDULING
ONTAP consente la gestione automatizzata degli Snapshot tramite policy:

  Policy Snapshot predefinita (esempio):
    Oraria:      mantieni 6 copie   (presa alle :00 di ogni ora)
    Giornaliera: mantieni 2 copie   (presa alle 00:10 ogni notte)
    Settimanale: mantieni 2 copie   (presa domenica alle 00:15)

Le policy personalizzate possono definire:
  - Qualsiasi schedule (stile cron: minuto, ora, giorno, giorno-settimana, mese)
  - Qualsiasi conteggio di retention (1-1023 per schedule)
  - Policy diverse per volumi diversi (produzione vs dev, database vs file share)

OPERAZIONI DI RESTORE BASATE SUGLI SNAPSHOT

Single file restore (NAS) — self-service utente:
L'utente naviga nella directory .snapshot/<nome_snapshot>/ e copia il file. Non è richiesto
l'intervento dell'amministratore.

Single file restore (admin):
  volume snapshot restore-file -vserver svm1 -volume vol1
    -snapshot hourly.2025-09-01_1200 -path /data/reports/report.xlsx
    -restore-path /data/reports/report_restored.xlsx

Full volume revert:
  volume snapshot restore -vserver svm1 -volume vol1
    -snapshot daily.2025-08-31_0010
  ATTENZIONE: tutte le modifiche dopo lo snapshot vengono permanentemente scartate


1.3 Relazioni di Data Protection
----------------------------------
La data protection ONTAP è costruita attorno al concetto di relazioni — connessioni logiche tra una
sorgente (primaria) e una destinazione (secondaria) che definiscono come e quando i dati vengono
replicati.

CONCETTI DELLE RELAZIONI
  Volume sorgente      - Il volume di produzione primario da proteggere
  Volume destinazione  - La copia secondaria (volume DP read-only durante la normale operazione)
  Tipo di relazione    - Definisce il comportamento di replica (async, sync, vault, ecc.)
  Policy               - Controlla la selezione degli snapshot, la retention sulla destinazione
                         e il throttle
  Schedule             - Quando vengono attivati gli aggiornamenti di replica (per async)
  Lag time             - Quanto indietro è la destinazione rispetto alla sorgente
  Stato healthy        - Se la relazione funziona normalmente

PEERING TRA CLUSTER E SVM
Prima che possa essere creata una relazione SnapMirror tra due cluster, i cluster devono essere
in peer:

  Peering cluster:
    cluster peer create -peer-addrs <IP-intercluster-destinazione> -ipspace Default
    (da eseguire su entrambi i cluster)

  Peering SVM (richiesto per relazioni di volume cross-cluster):
    vserver peer create -vserver svm_sorgente -peer-vserver svm_destinazione
      -peer-cluster cluster_destinazione -applications snapmirror

RIEPILOGO DEI TIPI DI RELAZIONE COMUNI
  XDP (Extended Data Protection) - SnapMirror Async / SnapVault - RPO: Min-Ore - Direzione: One-way
  DP (Data Protection)           - SnapMirror legacy (in dismissione) - RPO: Minuti - Direzione: One-way
  Sync                           - SnapMirror Synchronous - RPO: Zero - Direzione: One-way
  StrictSync                     - SnapMirror Sync (strict) - RPO: Zero - Direzione: One-way
  SM-BC / Active Sync            - SnapMirror Active Sync - RPO: Zero - Direzione: Active-Active


MODULE 2 - SOLUZIONI DI DISASTER RECOVERY
===========================================

2.1 Disaster Recovery: casi d'uso
-----------------------------------
Il Disaster Recovery (DR) si concentra sul ripristino delle operazioni dopo un evento che rende il
sistema primario non disponibile. Le soluzioni DR accettano una certa perdita di dati (RPO > 0) o
tempo di recovery (RTO > 0) in cambio di costi e complessità inferiori rispetto alle soluzioni di
business continuity.

Quando le soluzioni DR sono appropriate:
  Guasto array storage primario       - Failover al secondario SnapMirror sullo stesso sito
  Guasto data center primario (pianif.) - Failover SnapMirror schedulato verso sito remoto
  Ransomware o corruzione massiva dati - Failover a copia SnapMirror pulita prima dell'infezione
  Disastro regionale (alluvione, ecc.) - DR geografico verso regione remota
  Guasto o corruzione a livello SVM    - SVM DR con SnapMirror for SVM

Considerazioni per la pianificazione DR:
  - Distanza tra siti: latenza maggiore = RPO più alto per soluzioni sincrone
  - Banda tra siti: limita la velocità di replica rispetto al tasso di modifica
  - Tasso di modifica dati: workload ad alta modifica generano più traffico di replica
  - Consistency group: le applicazioni su più volumi richiedono la replica di tutti insieme
  - Automazione: il failover DR manuale è soggetto a errori; l'automazione riduce RTO ed errori


2.2 SnapMirror (Asincrono)
---------------------------
SnapMirror asincrono è la tecnologia principale di DR e replica dati di ONTAP. Replica gli Snapshot
da un volume sorgente a un volume destinazione su uno schedule.

PROCESSO DI REPLICA SNAPMIRROR

Passo 1 — Inizializzazione (trasferimento baseline):
  - Uno Snapshot completo del volume sorgente viene trasferito alla destinazione
  - Questo è l'unico trasferimento completo — tutti i successivi sono incrementali
  - La durata dipende dalla dimensione dei dati e dalla banda di rete

Passo 2 — Aggiornamenti incrementali:
  Su schedule (es. ogni ora), ONTAP:
    1. Crea un nuovo Snapshot sulla sorgente
    2. Identifica i blocchi modificati tra il nuovo snapshot e l'ultimo snapshot comune
    3. Trasferisce solo i blocchi modificati alla destinazione
    4. Aggiorna il volume di destinazione per riflettere il nuovo snapshot
    5. Elimina gli snapshot vecchi secondo la policy SnapMirror

Passo 3 — Failover (si verifica un disastro):
  snapmirror break -destination-path svm_dst:vol_dst
  (La destinazione diventa read-write; reindirizzare gli host alla destinazione)

Passo 4 — Resync (dopo il ripristino del primario):
  snapmirror resync -source-path svm_src:vol_src
    -destination-path svm_dst:vol_dst
  (Trasferisce solo i dati modificati di ritorno alla sorgente)

POLICY SNAPMIRROR
Le policy SnapMirror controllano quali Snapshot vengono replicati e quanti vengono conservati sulla
destinazione.

Policy integrate:
  MirrorAllSnapshots - Async Mirror  - Replica tutti gli snapshot sorgente alla destinazione
  MirrorLatest       - Async Mirror  - Replica solo l'ultimo snapshot
  MirrorAndVault     - Async Mix     - Combina mirror DR con retention vault a lungo termine
  Unified7year       - Vault         - Conserva gli snapshot per 7 anni (compliance)
  XDPDefault         - Async XDP     - Default per relazioni XDP

THROTTLING SNAPMIRROR
Controllo della banda consumata dalla replica per evitare impatti sulla produzione:
  snapmirror modify -destination-path svm_dst:vol_dst -throttle 102400
  (Il valore è in KB/s — questo esempio limita a ~100 MB/s)

FAN-OUT E CASCADE SNAPMIRROR

Fan-out: una sorgente replica verso più destinazioni simultaneamente
  Volume Sorgente --> Sito DR (SnapMirror Async)
                 --> Sito Backup (SnapVault)
                 --> Cloud (SnapMirror Cloud)

Cascade: la sorgente replica verso un intermediario, che replica verso la destinazione finale
  Produzione --> Sito DR --> Sito Archivio/Tape


2.3 SnapMirror for SVM DR
--------------------------
SVM DR estende SnapMirror per replicare non solo i volumi, ma l'intera configurazione SVM —
interfacce di rete, server CIFS, export NFS, igroup, utenti e tutti i volumi.

Cosa replica SVM DR:
  Tutti i volumi (dati)                - SI'
  Snapshot dei volumi                  - SI'
  Interfacce di rete SVM (LIF)         - SI' (IP preservati o rimappati)
  Configurazione server CIFS           - SI'
  Export policy NFS                    - SI'
  igroup iSCSI/FC e mappature LUN      - SI'
  Utenti e gruppi locali               - SI'
  Configurazione DNS e name service    - SI'
  Policy Snapshot a livello SVM        - SI'
  Chiavi di cifratura (NVE)            - SI' (se key manager in peer)

Failover SVM DR:
  snapmirror break -destination-path cluster_dst://svm_dst
  vserver start -vserver svm_dst
  (I client si riconnettono agli stessi indirizzi IP o tramite DNS aggiornato)

Casi d'uso SVM DR:
  - Migrazione completa SVM verso un nuovo cluster (migrazione non distruttiva)
  - Recovery completo SVM dopo corruzione o cancellazione accidentale
  - DR per ambienti multi-protocollo dove ricreare manualmente la configurazione richiederebbe
    ore


2.4 SnapMirror Synchronous
---------------------------
SnapMirror Synchronous replica ogni scrittura sia sulla sorgente che sulla destinazione prima di
confermare l'I/O all'applicazione host — garantendo RPO = 0 (zero perdita di dati).

Come funziona la replica sincrona:
  1. L'applicazione host invia una scrittura
  2. Il nodo ONTAP sorgente scrive sulla NVRAM locale
  3. Simultaneamente invia la scrittura alla destinazione via SnapMirror Sync
  4. Il nodo ONTAP destinazione scrive sulla propria NVRAM
  5. La destinazione invia un ACK alla sorgente
  6. La sorgente invia l'ACK all'host — SOLO dopo che ENTRAMBI i lati hanno confermato

Risultato: sorgente e destinazione sono SEMPRE sincronizzati (RPO = 0)

Modalità SnapMirror Synchronous:
  Sync       - In caso di interruzione rete, torna alla modalità async; continua a servire I/O
               RPO: quasi zero (breve finestra async)
  StrictSync  - Blocca gli I/O sulla sorgente se la destinazione diventa irraggiungibile
               RPO: zero assoluto — ma rischia l'indisponibilità degli I/O

La modalità Sync è raccomandata per la maggior parte dei workload di produzione — mantiene RPO zero
in condizioni normali ma non blocca la produzione se la connettività con il sito DR è brevemente
interrotta. StrictSync è riservata ai workload dove la perdita di dati è assolutamente inaccettabile
anche a costo della disponibilità (es. log transazioni finanziarie, cartelle cliniche).

Requisiti SnapMirror Synchronous:
  Latenza di rete:   <= 10 ms RTT tra sorgente e destinazione (idealmente <= 5 ms)
  Banda:             Deve accomodare il picco di scrittura del workload sorgente
  Versione ONTAP:    ONTAP 9.5+ per SnapMirror Synchronous
  Licenza:           Inclusa in ONTAP One
  Volumi supportati: Solo FlexVol (non FlexGroup)
  Max relazioni:     Fino a 80 relazioni sync per HA pair (ONTAP 9.9.1+)


2.5 SnapMirror S3
------------------
SnapMirror S3 estende la replica SnapMirror ai bucket S3 — abilitando la replica di dati object
storage tra bucket ONTAP S3 o tra ONTAP ed endpoint S3 cloud.

Casi d'uso SnapMirror S3:
  - S3 bucket DR: replica un bucket ONTAP S3 verso un altro cluster ONTAP per il DR
  - Backup S3 su cloud: replica dati ONTAP S3 verso AWS S3, Azure Blob o GCP GCS
  - Protezione applicazioni cloud-native: proteggi applicazioni che usano API S3

Schema architetturale SnapMirror S3:
  Bucket ONTAP S3 sorgente --> Bucket ONTAP S3 (su altro cluster)
                           --> Bucket AWS S3
                           --> Azure Blob Storage
                           --> Google Cloud Storage

Differenze chiave rispetto a SnapMirror per volumi:
  Unità di replica:   SnapMirror Volumi = volume + snapshot / SnapMirror S3 = oggetti S3
  Tipo destinazione:  SnapMirror Volumi = volume ONTAP (DP) / SnapMirror S3 = bucket S3
  Protocollo:         SnapMirror Volumi = nativo ONTAP / SnapMirror S3 = API S3
  Consistenza:        SnapMirror Volumi = basata su snapshot / SnapMirror S3 = eventual consistent
  Uso principale:     SnapMirror Volumi = DR volume, backup / SnapMirror S3 = replica object storage


MODULE 3 - SOLUZIONI DI BUSINESS CONTINUITY
=============================================

3.1 Business Continuity: casi d'uso
--------------------------------------
Le soluzioni di Business Continuity (BC) vanno oltre il disaster recovery: garantiscono che le
applicazioni continuino a servire I/O senza interruzioni anche durante un evento di guasto.
Le soluzioni BC puntano a RPO = 0 e RTO = 0 o quasi zero.

Quando le soluzioni BC sono richieste:
  Piattaforma di trading finanziario   - Anche secondi di downtime = milioni persi
  Cartelle cliniche pazienti           - I dati real-time devono essere sempre disponibili
  Sistemi core di telecomunicazioni    - L'indisponibilità impatta milioni di abbonati
  Checkout e-commerce nei picchi       - Qualsiasi downtime durante il Black Friday = perdita
  Sistemi di controllo manifatturieri  - Fermo impianto da guasto storage = costo enorme

BC vs DR — differenze chiave:
  RPO:                   DR = Minuti/Zero           / BC = Zero
  RTO:                   DR = Minuti-Ore            / BC = Zero-Secondi
  Downtime applicazione: DR = Sì (durante failover) / BC = No (trasparente)
  Costo:                 DR = Inferiore             / BC = Superiore
  Complessità:           DR = Media                 / BC = Alta
  Tecnologia esempio:    DR = SnapMirror Async      / BC = SnapMirror Active Sync, MetroCluster


3.2 SnapMirror Active Sync
---------------------------
SnapMirror Active Sync (ex SnapMirror Business Continuity / SM-BC) consente a due cluster ONTAP di
servire simultaneamente I/O per lo stesso dataset — fornendo RPO zero e RTO quasi zero per workload
SAN.

Architettura SnapMirror Active Sync:
  Sito A                                Sito B
  Cluster ONTAP A                       Cluster ONTAP B
  SVM-A --> Volume A (primario)         SVM-B --> Volume A' (mirror)
  Host A --> LUN (Sito A)               Host B --> LUN (Sito B)

  Entrambi gli host vedono lo STESSO LUN logico, servito dal loro sito locale.
  Le scritture vengono committed in modo sincrono su ENTRAMBI i siti prima dell'ACK.

  ONTAP Mediator (sito terzo o VM cloud)
    --> Risolve i pareggi durante una partizione di rete; attiva il failover automatico

Come funziona Active Sync:
  1. Entrambi i siti servono I/O di lettura localmente (dalla propria copia) — latenza minima
  2. Gli I/O di scrittura vengono committed in modo sincrono su entrambi i siti prima
     dell'ACK all'host
  3. Se il Sito A si guasta:
     - L'ONTAP Mediator rileva il guasto
     - Al Sito B viene concesso il diritto esclusivo di I/O
     - Il Sito B continua a servire tutti gli I/O — gli host al Sito B non subiscono interruzioni
     - Gli host al Sito A potrebbero richiedere riavvio o riconnessione

Capacità SnapMirror Active Sync:
  RPO:              Zero
  RTO:              Zero (per host al sito sopravvissuto); secondi per host al sito guasto
  Protocolli:       iSCSI, FC, NVMe/FC (solo SAN — non NAS)
  Volumi supportati: Solo FlexVol
  Host OS supportati: VMware vSphere, Windows, Linux, AIX, Solaris
  ONTAP Mediator:   Richiesto (tiebreaker per scenari split-brain)
  Distanza massima: Limitata solo dalla latenza di rete (<10ms RTT)
  Consistency group: Più volumi protetti come un'unica unità consistente

ONTAP MEDIATOR
L'ONTAP Mediator è un servizio software leggero deployato su un host terzo (fisico o VM)
indipendente da entrambi i siti di storage.

Ruolo:
  - Mantiene un canale di comunicazione continuo con entrambi i cluster ONTAP
  - Agisce come tiebreaker durante scenari split-brain (entrambi i siti attivi ma non
    comunicanti tra loro)
  - Se un sito si guasta, il Mediator conferma il guasto e consente al sito sopravvissuto di
    isolare il sito guasto e assumere la proprietà esclusiva degli I/O

Deployment:
  - Installato come applicazione Linux su una VM separata o server fisico
  - Deve essere raggiungibile da entrambi i cluster ma fuori da entrambi i siti primari
    (una VM cloud è valida)

CONSISTENCY GROUP (CG)
Per applicazioni che si estendono su più volumi (es. un database Oracle con dati, redo log e control
file su volumi separati), tutti i volumi devono essere protetti e sottoposti a failover insieme come
consistency group.

  consistency-group create -vserver svm1 -consistency-group app_cg
    -volumes vol_data,vol_logs,vol_control

  snapmirror create -source-path svm1:/cg/app_cg
    -destination-path svm2:/cg/app_cg_mirror -policy AutomatedFailOver


3.3 MetroCluster
-----------------
MetroCluster è la soluzione di business continuity hardware-level di NetApp che fornisce failover a
livello di sito con zero RPO e RTO quasi zero tramite mirroring hardware sincrono.

PANORAMICA DELL'ARCHITETTURA METROCLUSTER
MetroCluster opera a un livello fondamentalmente diverso da SnapMirror — esegue il mirroring a
livello RAID/aggregate, il che significa che ogni scrittura su ogni disco del Sito A viene
sincronicamente replicata al Sito B a livello hardware dello storage.

  SITO A                                     SITO B
  Cluster ONTAP A                            Cluster ONTAP B
  Node 1                                     Node 3
  Node 2  <--------- Mirror sincrono ------> Node 4
          (a livello hardware)
  Dischi locali + copie mirror dal Sito B    Dischi locali + copie mirror dal Sito A
                    |                                |
         ISL (Inter-Switch Link) FC o IP
                    |
         ONTAP Mediator (MetroCluster IP)

Configurazioni MetroCluster:
  MetroCluster FC 2 nodi  - 2 nodi, fabric FC                - Distanza: 300 km
  MetroCluster FC 4 nodi  - 2 HA pair (4 nodi totali)        - Distanza: 300 km
  MetroCluster FC 8 nodi  - 4 HA pair (8 nodi totali)        - Distanza: 300 km
  MetroCluster IP 2 nodi  - 2 nodi, rete IP                  - Distanza: 700 km (con DWDM)
  MetroCluster IP 4 nodi  - 2 HA pair (4 nodi totali)        - Distanza: 700 km
  MetroCluster IP 8 nodi  - 4 HA pair (8 nodi totali)        - Distanza: 700 km

Componenti chiave MetroCluster:
  Plex        - Ogni aggregate ha due copie (plex) — una per sito
  SyncMirror  - La tecnologia di mirroring che mantiene sincronizzati entrambi i plex
  ISL         - Link dedicato ad alta banda tra i due siti
  ONTAP Mediator - Richiesto per MetroCluster IP; fornisce switchover automatico
  Adapter FC-VI  - Richiesti per MetroCluster FC; forniscono cluster interconnect su FC

Switchover e Switchback MetroCluster:

Switchover negoziato (pianificato — es. per manutenzione del sito):
  metrocluster switchover
  (Il Sito B prende in carico tutti i workload; il Sito A va offline per manutenzione)

Switchover automatico (non pianificato):
  - Il Sito A si guasta completamente (perdita corrente, disastro)
  - L'ONTAP Mediator conferma che il Sito A è irraggiungibile
  - Il Sito B prende automaticamente in carico tutti i workload in ~30 secondi
  - Nessun intervento umano richiesto

Switchback (dopo il ripristino del Sito A):
  metrocluster heal -phase aggregates
  metrocluster heal -phase root-aggregates
  metrocluster switchback
  (Tutti i workload tornano al loro sito home)

MetroCluster vs SnapMirror Active Sync:
  Livello mirroring:       MetroCluster = Hardware (RAID/aggregate) / Active Sync = Software (volume)
  Protocolli:              MetroCluster = NAS + SAN (tutti)         / Active Sync = Solo SAN
  RPO:                     MetroCluster = Zero                       / Active Sync = Zero
  RTO:                     MetroCluster = ~30 secondi (auto)         / Active Sync = Quasi zero
  Distanza massima:        MetroCluster = 700 km (IP)                / Active Sync = Limitata da latenza
  Scala:                   MetroCluster = Fino a 8 nodi per config   / Active Sync = Per HA pair
  Consistenza:             MetroCluster = Intero cluster             / Active Sync = Consistency group
  Costo:                   MetroCluster = Più alto (HW dedicato)     / Active Sync = Inferiore (software)
  Ideale per:              MetroCluster = DC enterprise tutti protocolli / Active Sync = BC SAN-focused


3.4 Confronto tra soluzioni sincrone
--------------------------------------
  RPO:                   Sync = Zero / Active Sync = Zero / MetroCluster = Zero
  RTO:                   Sync = Minuti / Active Sync = Quasi zero / MetroCluster = ~30 sec
  Protocolli:            Sync = NAS+SAN / Active Sync = Solo SAN / MetroCluster = Tutti
  Trasparenza app:       Sync = No / Active Sync = Sì / MetroCluster = Sì
  I/O Active-Active:     Sync = No / Active Sync = Sì / MetroCluster = Sì
  ONTAP Mediator:        Sync = No / Active Sync = Sì / MetroCluster = Sì (IP)
  Max volumi/LUN:        Sync = 80/HA pair / Active Sync = 80 CG, 16 vol/CG / MetroCluster = Intero cluster
  Granularità:           Sync = Per volume / Active Sync = Per CG / MetroCluster = Per cluster
  Deployment tipico:     Sync = Singolo DC / Active Sync = Doppio DC SAN / MetroCluster = Doppio DC enterprise
  Licenza:               Sync = ONTAP One / Active Sync = ONTAP One / MetroCluster = Licenza separata


MODULE 4 - SOLUZIONI DI BACKUP E ARCHIVIO
===========================================

4.1 Backup e Archivio: casi d'uso
-----------------------------------
Mentre le soluzioni DR e BC proteggono contro i guasti del sito con RPO/RTO bassi, le soluzioni
di backup e archivio affrontano requisiti diversi:

  Scopo:       Backup = Ripristinare dati dopo perdita/corruzione
               Archivio = Conservare dati per compliance o accesso storico
  Retention:   Backup = Giorni-Mesi (recovery operativo)
               Archivio = Mesi-Anni (normativo, legale)
  Frequenza:   Backup = Occasionale (solo quando necessario)
               Archivio = Rara (recupero per audit o scoperta legale)
  RPO:         Backup = Ore-Giorni / Archivio = Ore-Giorni
  RTO:         Backup = Ore / Archivio = Ore-Giorni
  Immutabilità: Backup = Raccomandata / Archivio = Spesso richiesta (WORM)

Backup vs Snapshot vs Replica:
  Posizione:              Snapshot = Stesso sistema / SnapMirror = Sistema remoto / Backup = Sistema separato o cloud
  Protegge da guasto HW?: Snapshot = No / SnapMirror = Sì / Backup = Sì
  Protegge da ransomware: Snapshot = Parziale / SnapMirror = Parziale / Backup = Sì (copia indipendente)
  Durata retention:       Snapshot = Ore-Settimane / SnapMirror = Giorni-Settimane / Backup = Mesi-Anni
  Granularità restore:    Snapshot = File, LUN, vol / SnapMirror = Volume, SVM / Backup = File, LUN, vol
  Indipendenza sorgente:  Snapshot = No / SnapMirror = Parziale / Backup = Completa


4.2 SnapMirror per Backup e Archivio (SnapVault)
-------------------------------------------------
SnapMirror in modalità XDP (Extended Data Protection) con una policy vault fornisce la capacità
nativa di backup disk-to-disk di ONTAP (precedentemente nota come SnapVault).

ARCHITETTURA SNAPVAULT:
  Cluster Produzione                    Cluster Backup/Vault
  Volume Produzione                     Volume Vault (DP, read-only)
  Snapshot conservati:                  Snapshot conservati (indipendentemente):
  - orari x 6                           - giornalieri x 30
  - giornalieri x 2                     - settimanali x 52
  - settimanali x 2                     - mensili x 24

  ---- SnapMirror XDP (vault policy) --->

COMPORTAMENTI CHIAVE DI SNAPVAULT
  - Indipendenza dalla sorgente: gli snapshot conservati nel vault sono indipendenti dalla
    sorgente. Cancellare uno snapshot sulla sorgente NON lo cancella dal vault.
  - Retention a lungo termine: le policy vault possono conservare snapshot per settimane, mesi
    o anni.
  - Trasferimento efficiente: vengono trasferiti solo i blocchi modificati (non copie complete).
  - Cascade vault-to-vault: una copia vault può essere replicata verso un secondo vault per
    protezione aggiuntiva.

Configurazione policy SnapVault:
  snapmirror policy create -policy vault_policy -type vault
  snapmirror policy add-rule -policy vault_policy -snapmirror-label daily -keep 30
  snapmirror policy add-rule -policy vault_policy -snapmirror-label weekly -keep 52
  snapmirror policy add-rule -policy vault_policy -snapmirror-label monthly -keep 24

  snapmirror create -source-path svm_src:vol_src
    -destination-path svm_vault:vol_vault
    -policy vault_policy -schedule daily

Ripristino da SnapVault:

  Ripristino singolo file dal vault alla produzione:
    snapmirror restore -source-path svm_vault:vol_vault
      -destination-path svm_src:vol_src
      -source-snapshot weekly.2025-08-31_0015
      -file-list /data/finance/q2_report.xlsx

  Ripristino intero volume:
    snapmirror restore -source-path svm_vault:vol_vault
      -destination-path svm_src:vol_recuperato
      -source-snapshot monthly.2025-08-01_0015


4.3 SnapCenter
---------------
SnapCenter è la piattaforma di backup e recovery enterprise di NetApp — una soluzione di backup
centralizzata e application-aware costruita sulle tecnologie ONTAP Snapshot, SnapMirror e
SnapVault.

COSA FA SNAPCENTER
SnapCenter estende la protezione dati nativa di ONTAP aggiungendo:
  - Application awareness: mette in quiesce le applicazioni prima di prendere snapshot (garantisce
    consistenza)
  - Gestione centralizzata: unica UI per gestire le policy di backup su tutte le applicazioni e
    lo storage
  - Self-service restore: i proprietari delle applicazioni possono ripristinare i propri dati
    senza coinvolgere l'amministratore storage
  - Catalogo e ricerca: catalogo backup ricercabile per recovery rapido
  - Automazione policy: schedule di backup automatizzati, retention e verifica
  - Integrazione SnapMirror/SnapVault: replica/vault automaticamente gli snapshot di backup verso
    storage secondario o cloud

Architettura SnapCenter:
  SnapCenter Server (Windows)
    +-- UI SnapCenter (browser web)
    +-- REST API SnapCenter
    +-- Job scheduler e policy engine
    +-- Plugin ONTAP (via ONTAP REST API) --> Cluster ONTAP
    +-- Plugin applicativi:
        +-- SnapCenter Plugin for VMware vSphere
        +-- SnapCenter Plugin for Oracle Database
        +-- SnapCenter Plugin for SQL Server
        +-- SnapCenter Plugin for SAP HANA
        +-- SnapCenter Plugin for Exchange
        +-- SnapCenter Plugin for Windows (filesystem)
        +-- SnapCenter Plugin for Unix/Linux (filesystem)
    +-- Agenti SnapCenter (installati sugli host applicativi)

PROCESSO DI BACKUP APPLICATION-CONSISTENT
Esempio: backup SQL Server con SnapCenter
  1. L'agente SnapCenter sull'host SQL Server segnala a SQL di bloccare gli I/O (VSS quiesce)
  2. ONTAP crea uno Snapshot application-consistent di tutti i volumi SQL simultaneamente
  3. SQL Server viene sbloccato (I/O riprende) — tempo totale di freeze: tipicamente <1 secondo
  4. SnapCenter cataloga il backup con metadata (istanza SQL, database, timestamp)
  5. SnapCenter attiva l'aggiornamento SnapMirror/SnapVault per replicare lo snapshot al
     secondario

Tipi di recovery SnapCenter:
  Volume-level restore - Ripristina intero volume da snapshot                 - Perdita massiva, corruzione volume
  Database restore     - Ripristina file specifici del database                - Corruzione singolo database
  Table/item restore   - Restore granulare di tabelle o elementi specifici     - Cancellazione accidentale dati
  VM restore           - Ripristina intera VM o file singoli (VMware)          - Corruzione o cancellazione VM
  Clone                - Crea FlexClone da backup per testing                  - Test/dev, verifica backup

SnapCenter vs SnapMirror/SnapVault nativo:
  Consistenza applicativa:  Nativo = Manuale (script) / SnapCenter = Automatica (plugin integrati)
  Gestione centralizzata:   Nativo = No (per cluster) / SnapCenter = Sì (unica interfaccia)
  Self-service restore:     Nativo = No               / SnapCenter = Sì (tramite UI SnapCenter)
  Catalogo backup:          Nativo = No               / SnapCenter = Sì (ricercabile)
  Plugin applicativi:       Nativo = No               / SnapCenter = Sì (Oracle, SQL, VMware, HANA, ecc.)
  Recovery granulare:       Nativo = Volume/file       / SnapCenter = A livello applicazione
  Licenza:                  Nativo = Incluso ONTAP One / SnapCenter = Licenza SnapCenter separata


MODULE 5 - DR E BACKUP SU CLOUD
=================================

5.1 Backup su Cloud: casi d'uso
---------------------------------
La protezione dati basata su cloud è diventata un componente essenziale delle moderne
strategie di data protection:

  Protezione offsite    - Il cloud è geograficamente separato dall'on-premises per definizione
  Efficienza costi      - L'object storage cloud è significativamente più economico dello storage
                          secondario on-premises
  Scalabilità           - Nessuna pianificazione della capacità necessaria — il cloud cresce
                          on-demand
  Semplicità            - Nessun hardware secondario da acquistare, installare e gestire
  Capacità air-gap      - Le copie cloud possono essere isolate dall'on-premises (limitando la
                          portata del ransomware)
  Accessibilità globale - Recovery possibile da qualsiasi posizione connessa a Internet

Casi d'uso protezione cloud per scenario:
  Copia backup offsite         - SnapMirror Cloud / BlueXP Backup
  DR verso cloud               - Cloud Volumes ONTAP come target DR
  Backup applicazioni cloud    - BlueXP Backup per FSx / Azure NetApp Files
  Archivio a lungo termine     - Object storage cloud con tier Glacier/Archive
  Air-gap ransomware           - S3 object lock immutabile sulle copie di backup


5.2 Integrazione con Software di Terze Parti
---------------------------------------------
La REST API aperta di ONTAP e i protocolli standard abilitano l'integrazione con le principali
piattaforme di backup e protezione dati di terze parti.

Integrazioni comuni:
  Veeam Backup & Replication  - ONTAP REST API + SnapMirror - Backup VM con Snapshot ONTAP; instant VM recovery
  Commvault                   - ONTAP SnapMirror + API      - Backup enterprise per ONTAP NAS/SAN + cloud
  Veritas NetBackup           - ONTAP API + NDMP            - Backup enterprise, integrazione tape
  Cohesity DataProtect        - ONTAP Snapshot API          - Backup moderno per applicazioni enterprise
  Rubrik                      - ONTAP API                   - Backup e recovery cloud-first
  IBM Spectrum Protect        - ONTAP NDMP + API            - Backup enterprise + tape
  Zerto                       - Replica continua + ONTAP    - Protezione dati continua e DR cloud

NDMP (NETWORK DATA MANAGEMENT PROTOCOL)
NDMP è un protocollo standard che consente alle applicazioni di backup di comunicare
direttamente con i sistemi storage NAS per operazioni efficienti di backup e restore:
  - L'applicazione di backup invia comandi NDMP a ONTAP
  - ONTAP trasmette i dati direttamente al target di backup (tape, VTL o disco)
  - NDMP three-way: ONTAP → media server backup → tape library
  - NDMP two-way (local): ONTAP trasmette direttamente al dispositivo tape collegato

INTEGRAZIONE VEEAM + ONTAP
Veeam fornisce una delle integrazioni più profonde con ONTAP:
  - Snapshot ONTAP per consistenza VM: Veeam può attivare Snapshot ONTAP nel workflow di backup VM
  - Integrazione SnapMirror: Veeam può usare le repliche SnapMirror come sorgenti di backup
    (riduce l'impatto sulla produzione)
  - Instant VM Recovery da ONTAP: avvia VM direttamente da Snapshot/clone ONTAP — nessuna
    copia dati necessaria
  - NAS Backup: Veeam esegue il backup di share NAS ONTAP con granularità a livello di file
  - SureBackup: verifica automatica del backup usando ambienti di test basati su FlexClone


5.3 Cloud Volumes ONTAP (CVO)
------------------------------
Cloud Volumes ONTAP è il software NetApp ONTAP deployato come appliance storage virtuale nei cloud
pubblici (AWS, Azure, GCP). Abilita le funzionalità ONTAP complete nel cloud, rendendolo ideale come
target DR cloud-based.

CLOUD VOLUMES ONTAP COME TARGET DR
  On-prem ONTAP (AFF/FAS)               Cloud Volumes ONTAP (CVO)
  SVM sorgente / Volume                  SVM destinazione / Volume
         |                                         |
         +----------- SnapMirror Async ----------->+
                      (via Internet o Direct Connect)

  In caso di failover:
    - Interrompi SnapMirror su CVO
    - Attiva CVO come produzione
    - Monta via NFS/SMB/iSCSI da host cloud o host on-prem connessi via VPN

Opzioni di deployment CVO:
  Singolo nodo     - 1 istanza CVO    - Dev/test, workload non critici, ottimizzato per costi
  HA pair          - 2 istanze (A-A)  - Workload produzione con requisiti HA
  Multi-AZ HA      - 2 istanze in AZ diverse - Massima disponibilità nel cloud

Funzionalità chiave CVO per la data protection:
  - Compatibilità SnapMirror completa: CVO è un peer ONTAP completo — SnapMirror funziona
    identicamente tra ONTAP on-prem e CVO
  - Integrazione FabricPool: tier dati freddi da CVO verso object storage cloud nativo
  - Gestione BlueXP: gestisci ONTAP on-prem e CVO dalla stessa interfaccia BlueXP
  - Licenze ONTAP One: applica le licenze ONTAP One esistenti a CVO (modello BYOL) o usa
    pay-as-you-go
  - Cifratura NVE: dati a riposo cifrati tramite ONTAP Volume Encryption

Ottimizzazione costi per DR CVO:
  - Esegui CVO alla dimensione minima di istanza durante le normali operazioni (solo ricevendo
    aggiornamenti SnapMirror)
  - Scala su l'istanza CVO al momento del failover per corrispondere ai requisiti di produzione
  - Usa il pricing reserved instance del provider cloud per costi prevedibili


5.4 BlueXP Backup and Recovery
--------------------------------
BlueXP Backup and Recovery (ex Cloud Backup) è la soluzione di backup-as-a-service cloud-native di
NetApp, integrata direttamente nella piattaforma di gestione BlueXP.

COSA FA BLUEXP BACKUP AND RECOVERY
BlueXP Backup and Recovery fornisce backup policy-based dei volumi ONTAP direttamente verso object
storage cloud, senza richiedere un sistema ONTAP secondario.

  SORGENTE                               DESTINAZIONE (Cloud Object Storage)
  Cluster ONTAP on-premises     ----->   AWS S3
  Cloud Volumes ONTAP           ----->   Azure Blob Storage
  Amazon FSx for NetApp ONTAP   ----->   Google Cloud Storage
  Azure NetApp Files            ----->   StorageGRID (object storage on-prem)

Funzionalità chiave BlueXP Backup:
  Scheduling policy-based    - Definisci frequenza backup e retention
  Incremental forever        - Dopo il backup iniziale completo, solo i blocchi modificati
                               vengono salvati (efficiente in spazio/banda)
  Compressione e dedup       - Applicata prima dell'invio al cloud — riduce i costi
  Cifratura end-to-end       - TLS in transito + AES-256 a riposo nell'object storage
  Object lock / immutabilità  - Integrazione con AWS S3 Object Lock o Azure Immutable Blob
  DataLock e protezione ARP  - Layer di immutabilità gestito da BlueXP per le copie di backup
  Ricerca nel catalogo        - Cerca file specifici in tutte le copie di backup
  Restore granulare file      - Ripristina singoli file o directory da qualsiasi copia
  Restore volume              - Ripristina interi volumi sullo stesso o diverso sistema
  Restore cross-region        - Ripristina dati in una diversa regione cloud
  Cost tiering                - Sposta automaticamente le copie più vecchie verso tier cloud
                               più economici (S3 Glacier, Azure Cool/Archive)

BlueXP Backup Policy — esempio:
  Policy: "production_backup_policy"
  Oraria:    24 copie  --> conservate in tier "Standard"
  Giornaliera: 30 copie --> conservate in tier "Standard"
  Settimanale: 13 copie --> spostate in "Infrequent Access" dopo 30 giorni
  Mensile:   12 copie  --> spostate in "Archive" dopo 90 giorni
  Annuale:    7 copie  --> conservate in tier "Archive"
              DataLock: abilitato (immutabile, protetto da ransomware)

PROTEZIONE RANSOMWARE CON BLUEXP BACKUP
BlueXP si integra con ONTAP Autonomous Ransomware Protection (ARP) e aggiunge il proprio
layer:
  1. ARP rileva attività sospette su un volume sorgente → crea uno snapshot immutabile
  2. Il backup BlueXP di quello snapshot viene inviato all'object storage cloud con DataLock
     abilitato
  3. La copia di backup cloud è immutabile — non può essere cancellata dal ransomware anche
     se le credenziali cloud sono compromesse
  4. Recovery: ripristino dall'ultima copia di backup pulita nel cloud, completamente
     indipendente dall'infrastruttura on-premises


5.5 Scelta della giusta soluzione di Data Protection
------------------------------------------------------

FRAMEWORK DECISIONALE

OBIETTIVO: BACKUP/ARCHIVIO
  Hai bisogno di backup application-consistent?
    Sì --> SnapCenter (con plugin specifici per l'applicazione)
    No  --> Policy vault SnapMirror nativo o BlueXP Backup
  Dove si trova il target di backup?
    On-premises secondario --> SnapVault (SnapMirror XDP vault)
    Cloud                  --> BlueXP Backup and Recovery
  È richiesta compliance/WORM a lungo termine?
    Sì --> SnapLock Enterprise/Compliance + BlueXP DataLock
    No  --> Retention vault standard

OBIETTIVO: DISASTER RECOVERY
  Qual è il requisito RPO?
    Ore     --> SnapMirror Async (schedule standard XDP)
    Minuti  --> SnapMirror Async (schedule frequente)
    Zero    --> SnapMirror Synchronous o Active Sync
  Il target DR è on-premises o cloud?
    On-premises --> SnapMirror verso ONTAP secondario
    Cloud       --> Cloud Volumes ONTAP + SnapMirror
  L'intero SVM deve essere recuperabile?
    Sì --> SnapMirror for SVM DR
    No  --> SnapMirror a livello di volume

OBIETTIVO: BUSINESS CONTINUITY
  Quali protocolli devono rimanere disponibili?
    Solo SAN (iSCSI/FC/NVMe) --> SnapMirror Active Sync
    Tutti i protocolli (NAS + SAN) --> MetroCluster
  Qual è l'RTO accettabile?
    Quasi zero (trasparente) --> SnapMirror Active Sync o MetroCluster
    Minuti                   --> SnapMirror Synchronous
  Quale distanza è richiesta tra i siti?
    Qualsiasi distanza (<10ms RTT) --> SnapMirror Active Sync
    Fino a 700 km (HW dedicato)    --> MetroCluster IP

MATRICE DI CONFRONTO COMPLETA
  Snapshot               - RPO: Variabile  - RTO: Secondi     - Protocolli: Tutti - Scope: Vol/file   - Tipo: Backup
  SnapRestore            - RPO: Per snap   - RTO: Secondi     - Protocolli: Tutti - Scope: File/LUN/vol - Tipo: Recovery backup
  SnapMirror Async       - RPO: Minuti     - RTO: Min-Ore     - Protocolli: Tutti - Scope: Vol/SVM    - Tipo: DR
  SnapMirror Sync        - RPO: Zero       - RTO: Minuti      - Protocolli: NAS+SAN - Scope: Volume   - Tipo: DR
  SnapMirror Active Sync - RPO: Zero       - RTO: Quasi zero  - Protocolli: SAN   - Scope: CG/Volume  - Tipo: BC
  SnapVault              - RPO: Ore        - RTO: Ore         - Protocolli: Tutti - Scope: Volume     - Tipo: Backup
  MetroCluster           - RPO: Zero       - RTO: ~30 sec     - Protocolli: Tutti - Scope: Cluster    - Tipo: BC
  SnapCenter             - RPO: App-cons.  - RTO: App-level   - Protocolli: Tutti - Scope: App/VM     - Tipo: Backup
  Cloud Volumes ONTAP    - RPO: Minuti     - RTO: Min-Ore     - Protocolli: Tutti - Scope: Vol/SVM    - Tipo: DR cloud
  BlueXP Backup          - RPO: Ore        - RTO: Ore         - Protocolli: Tutti - Scope: Vol/File   - Tipo: Backup/Archivio
  SnapMirror S3          - RPO: Minuti     - RTO: Minuti      - Protocolli: S3    - Scope: Bucket     - Tipo: DR/Backup
