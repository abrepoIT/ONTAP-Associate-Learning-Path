ONTAP CLUSTER FUNDAMENTALS
===========================


MODULE 1 - UNDERSTANDING ONTAP CLUSTERS
=========================================

1.1 Cos'è un Cluster ONTAP?
-----------------------------
Un cluster ONTAP è una raccolta di uno o più nodi di storage che eseguono il software NetApp ONTAP e
lavorano insieme come un unico sistema di storage unificato. Dal punto di vista di un client, un
cluster appare come un'unica entità logica — indipendentemente da quanti nodi fisici contiene.

Il clustering in ONTAP fornisce:
  - Scalabilità: aggiunta di nodi per aumentare capacità e performance senza interruzioni
  - Alta disponibilità: più nodi proteggono contro i guasti dei singoli nodi
  - Operazioni non distruttive: manutenzione hardware, upgrade e bilanciamento dei workload
    possono essere eseguiti mentre le applicazioni rimangono online
  - Gestione unificata: una singola interfaccia di gestione governa tutti i nodi e le risorse

Dimensione del cluster:
  AFF A-Series / C-Series - Min: 1 nodo - Max: 24 nodi - Note: Cluster all-flash scale-out
  FAS                     - Min: 1 nodo - Max: 24 nodi - Note: Storage ibrido
  ASA                     - Min: 1 nodo - Max: 12 nodi - Note: Ottimizzato per SAN
  ONTAP Select            - Min: 1 nodo - Max: 4 nodi  - Note: Software-defined su HW commodity
  Cloud Volumes ONTAP     - Min: 1 nodo - Max: 2 (HA)  - Note: AWS, Azure, GCP


1.2 Storage Node
-----------------
Uno storage node (semplicemente chiamato "nodo") è un singolo storage controller — un server
fisico o virtuale che esegue il software ONTAP. Ogni nodo contribuisce con:
  - CPU e memoria (per l'elaborazione ONTAP, il caching e il RAID)
  - Drive di storage (HDD, SSD, NVMe) organizzati in disk shelf
  - Porte di rete (per l'accesso ai dati e la comunicazione del cluster)
  - NVRAM / NVDIMM (write cache non volatile)

Identità del nodo:
Ogni nodo in un cluster ha:
  - Un nome di nodo univoco (es. cluster1-01, cluster1-02)
  - Un UUID di nodo univoco
  - Un system ID usato internamente da ONTAP
  - Un indirizzo IP di gestione per l'accesso out-of-band

Processo di boot del nodo:
Quando un nodo si avvia, ONTAP:
  1. Carica il kernel ONTAP dal dispositivo di boot
  2. Esegue auto-test hardware
  3. Si unisce al cluster (o ne forma uno nuovo se parte da zero)
  4. Riproduce la NVRAM (write cache) per garantire la consistenza dopo uno shutdown non pulito
  5. Monta gli aggregate e porta i volumi online


1.3 HA Pair (High Availability Pair)
--------------------------------------
Un HA pair è un insieme di due nodi configurati per fornire protezione reciproca tramite failover. È
il building block fondamentale di un cluster ONTAP.

Come funziona l'HA:
  Node 1 (Primary)  <------ HA Interconnect ------>  Node 2 (Partner)
  - Possiede Aggr1            (NVRAM mirroring           - Possiede Aggr3
  - Possiede Aggr2             + heartbeat)              - Possiede Aggr4
                    <-------- Shared Disk Shelves ------->
                           (ogni nodo possiede i propri drive)

HA Interconnect:
  - Una connessione ad alta velocità dedicata tra i due nodi
  - Replica la write cache NVRAM di ogni nodo sul suo partner in tempo reale
  - Invia messaggi heartbeat periodici per rilevare i guasti del nodo

Takeover (Failover):
Quando il Node 1 si guasta (guasto hardware, panic, perdita di alimentazione):
  1. Il Node 2 rileva il guasto tramite heartbeat mancati
  2. Il Node 2 esegue un takeover: assume la proprietà degli aggregate e delle LIF del Node 1
  3. Il Node 2 serve gli I/O sia per i propri che per i workload del Node 1
  4. I client subiscono un'interruzione minima (tipicamente secondi)

Giveback:
Quando il Node 1 viene ripristinato:
  1. Il Node 1 si avvia e si riconnette al cluster
  2. Il Node 2 esegue un giveback: restituisce gli aggregate e le LIF del Node 1
  3. Entrambi i nodi riprendono la normale operazione

Configurazioni dell'HA Pair:
  Active-Active        - Entrambi i nodi servono I/O client simultaneamente (operazione normale)
  Takeover state       - Un nodo serve tutti gli I/O dopo il guasto del partner
  Waiting for giveback - Il nodo guasto è ripristinato e in attesa che le risorse vengano restituite

Operazioni non distruttive abilitate dall'HA:
  - Planned takeover: l'amministratore attiva manualmente un takeover per eseguire manutenzione
    su un nodo (aggiornamento firmware, sostituzione hardware) senza impattare i client
  - Automatic failover: takeover non pianificato attivato da guasto del nodo
  - LIF failover: le interfacce di rete (LIF) migrano al nodo partner durante il takeover


1.4 L'Architettura del Cluster ONTAP
--------------------------------------
Un cluster è organizzato in una gerarchia di componenti logici e fisici:

  ONTAP CLUSTER
  |
  +-- Node 1 (HA Pair con Node 2)
  |   +-- NVRAM (write cache, mirrored su Node 2)
  |   +-- Aggregate 1
  |   |   +-- RAID Group 1 (drive)
  |   |   +-- RAID Group 2 (drive)
  |   +-- Aggregate 2
  |       +-- RAID Group 3 (drive)
  |
  +-- Node 2 (HA Pair con Node 1)
  |   +-- NVRAM (write cache, mirrored su Node 1)
  |   +-- Aggregate 3
  |       +-- RAID Group 4 (drive)
  |
  +-- Node 3 (HA Pair con Node 4)
  |   +-- Aggregate 4 ...
  |
  +-- Node 4 (HA Pair con Node 3)
      +-- Aggregate 5 ...

Cluster Quorum:
ONTAP usa un meccanismo di quorum per garantire la consistenza del cluster:
  - Il quorum determina quali nodi sono membri autorevoli del cluster
  - Un nodo deve essere "in quorum" per servire i dati
  - Se un nodo perde il contatto con il cluster, si isola per prevenire la corruzione dei dati
    (protezione split-brain)
  - Epsilon: un peso di voto speciale assegnato a un nodo per rompere i pareggi nei cluster
    con numero pari di nodi


MODULE 2 - ARCHITETTURA DI STORAGE ONTAP
==========================================

2.1 Componenti di storage fisici
----------------------------------

DISK SHELF
I disk shelf (detti anche drive shelf o JBOD) sono gli involucri fisici che contengono i drive di
storage e si connettono ai nodi ONTAP.

Componenti chiave degli shelf:
  - Drive bay: slot che contengono i singoli drive (HDD, SSD, NVMe)
  - Moduli I/O (IOM): controller all'interno dello shelf che gestiscono la comunicazione
    con i nodi
  - Alimentatori: ridondanti per alta disponibilità
  - Ventole: unità di raffreddamento, tipicamente anch'esse ridondanti

Connettività degli shelf:
  - Shelf SAS: si connettono ai nodi tramite cavi SAS (Serial Attached SCSI); supportano HDD
    e SSD SAS
  - Shelf NVMe: si connettono tramite NVMe over SAS o PCIe; supportano SSD NVMe per latenza
    ultra-bassa

Shelf ID: ogni shelf in uno stack riceve un ID univoco (0-98) per distinguerlo dagli altri shelf
nello stesso dominio SAS.

Tipi di drive supportati da ONTAP:
  HDD (NL-SAS) - Interfaccia: SAS   - Uso: Storage secondario ad alta capacità, archivio - Latenza: 5-10 ms
  SAS SSD      - Interfaccia: SAS   - Uso: Tier performance, all-flash                   - Latenza: ~100 µs
  NVMe SSD     - Interfaccia: NVMe  - Uso: Ultra-alta performance, piattaforme AFF        - Latenza: 20-100 µs

Spare Drive e Hot Spare:
  - ONTAP richiede almeno un drive spare per tipo di RAID group (per nodo)
  - Hot spare: un drive non allocato che ONTAP usa automaticamente per ricostruire il RAID
    group di un drive guasto
  - I drive spare devono essere dello stesso tipo e di dimensione uguale o superiore ai drive
    che proteggono


2.2 Componenti di storage logici ONTAP
----------------------------------------
ONTAP presenta lo storage attraverso una gerarchia di oggetti logici. Comprendere questa gerarchia è
essenziale per gestire i sistemi ONTAP.

  LAYER FISICO               LAYER LOGICO
  Drive (HDD/SSD)       -->  RAID Group
                        -->  Aggregate (contiene RAID group)
                        -->  Volume (FlexVol/FlexGroup — dentro gli aggregate)
                        -->  Qtree (partizioni di directory opzionali nei volumi)
                        -->  LUN (oggetti block storage nei volumi)
                        -->  File (dati NAS nei volumi)

RAID GROUP
Un RAID group è il raggruppamento logico di livello più basso di drive fisici in ONTAP. ONTAP usa il
RAID per proteggere i dati contro i guasti dei drive.

Tipi di RAID in ONTAP:
  RAID4     - 1 parity dedicata  - Tolleranza: 1 drive  - Dimensione gruppo: Legacy, non raccomandato
  RAID-DP   - 2 parity distr.    - Tolleranza: 2 drive  - Dimensione gruppo: 12-28 drive (default)
  RAID-TEC  - 3 parity distr.    - Tolleranza: 3 drive  - Dimensione gruppo: 20-28 drive (per drive >8TB)

RAID-DP (Double Parity):
  - Implementazione RAID-6 proprietaria di NetApp
  - Due stripe di parità indipendenti proteggono contro due guasti simultanei di drive
  - Il tipo RAID predefinito per tutti i deployment ONTAP
  - Raccomandato per HDD NL-SAS dove i tempi di rebuild sono lunghi

RAID-TEC (Triple Erasure Coding):
  - Aggiunge una terza stripe di parità per una protezione ancora maggiore
  - Progettato per drive molto grandi (8 TB+) dove i rebuild RAID richiedono molte ore
  - Protegge contro tre guasti simultanei di drive durante un rebuild

AGGREGATE
Un aggregate è una raccolta di uno o più RAID group che forma un pool di storage fisico. È assegnato
a un nodo specifico.

Caratteristiche chiave:
  - Un aggregate appartiene esattamente a un nodo (o viene preso in carico dal partner in caso
    di failover)
  - Tutti i drive in un aggregate devono essere dello stesso tipo (solo SSD o solo HDD — no mix)
  - Un aggregate può contenere più RAID group
  - I volumi vengono creati all'interno degli aggregate

Tipi di aggregate:
  - HDD aggregate: contiene HDD NL-SAS o SAS → orientato alla capacità
  - SSD aggregate: contiene SSD SAS o NVMe → orientato alle performance
  - Flash Pool (hybrid aggregate): combina HDD con un tier cache SSD per performance migliorate
    a costo inferiore rispetto all'all-flash

Root aggregate:
  - Ogni nodo ha un root aggregate dedicato contenente il root volume del nodo
  - Il root volume contiene i file di configurazione, i log e i dati di sistema di ONTAP
  - Il root aggregate è tipicamente piccolo (20-50 GB) e separato dagli aggregate dati

VOLUME
Un volume è il contenitore logico fondamentale per i dati in ONTAP. I volumi risiedono all'interno
degli aggregate e vengono presentati ai client come filesystem (NAS) o come contenitori per LUN
(SAN).

Tipi di volume:
  FlexVol             - Volume ONTAP standard; risiede in un aggregate; fino a 300 TB
  FlexGroup           - Volume distribuito su più aggregate/nodi; fino a 20 PB
  FlexCache           - Volume read-cache supportato da un volume di origine su un altro cluster
  SnapLock            - Volume WORM immutabile per compliance o protezione ransomware
  Data Protection (DP) - Volume di destinazione read-only per la replica SnapMirror
  Root volume         - Volume di sistema contenente la configurazione del nodo ONTAP

Stati del volume:
  - Online: accessibile ai client
  - Offline: non accessibile (portato offline manualmente o dopo un errore)
  - Restricted: accessibile solo per certe operazioni amministrative

Garanzie del volume (prenotazione spazio):
  Volume - La capacità completa del volume è riservata dall'aggregate al momento della creazione
  None (thin) - Nessuno spazio è riservato; lo spazio fisico viene consumato man mano che i
                dati vengono scritti

Junction path (namespace NAS):
  - Ogni volume NAS viene montato in un junction path nel namespace dell'SVM
  - Esempio: un volume chiamato vol_data montato in /data nel namespace dell'SVM
  - I client accedono ai dati montando il root volume dell'SVM e navigando verso /data

QTREE
Un qtree è una partizione logica opzionale all'interno di un volume FlexVol, simile a una
directory ma con capacità di gestione aggiuntive:
  - Applicare export policy separate (controlli di accesso NFS) per qtree
  - Applicare permessi a livello di share separati (SMB) per qtree
  - Applicare quote individuali (limitare lo spazio disco o il numero di file per utenti/gruppi)
  - Fornire un ulteriore layer di organizzazione del namespace

LUN (LOGICAL UNIT NUMBER)
Un LUN è un oggetto di block storage logico presentato agli host SAN come un disco virtuale.

Ciclo di vita di un LUN:
  1. Creare un volume (con prenotazione spazio o thin provisioning)
  2. Creare un LUN all'interno del volume (specificare dimensione e tipo OS)
  3. Creare un igroup (initiator group) contenente gli initiator dell'host (iSCSI IQN o FC WWPN)
  4. Mappare il LUN all'igroup
  5. L'host scopre il LUN come un nuovo dispositivo disco

Tipi OS per i LUN: ONTAP usa il tipo OS per determinare l'allineamento dei blocchi e altri
parametri: linux, windows, windows_2008, vmware, xen, hyper_v, solaris, aix, hpux

Gestione spazio LUN:
  - Space-reserved LUN: la dimensione completa del LUN è pre-allocata nel volume
  - Thin-provisioned LUN: lo spazio viene consumato man mano che i dati vengono scritti

STORAGE VIRTUAL MACHINE (SVM)
Un SVM (Storage Virtual Machine, detto anche Vserver) è un'entità logica di data-serving all'interno
di un cluster ONTAP. Gli SVM forniscono multi-tenancy e flessibilità dei protocolli.

Tipi di SVM:
  Data SVM   - Serve i dati client via protocolli NAS e/o SAN; il tipo di SVM principale
  Admin SVM  - Rappresenta il cluster stesso; usato per la gestione a livello di cluster
  Node SVM   - Rappresenta un singolo nodo; usato per la gestione a livello di nodo
  System SVM - Usato internamente per la comunicazione del cluster

Componenti di una Data SVM:
  Volumi         - Contenitori dati di proprietà dell'SVM
  LIF            - Interfacce di rete (indirizzi IP o FC WWPN) per l'accesso ai dati
  Namespace      - L'albero di directory dell'SVM (junction path per i volumi NAS)
  Protocolli     - I protocolli serviti dall'SVM (NFS, SMB, iSCSI, FC, NVMe/FC, S3)
  Export policy  - Regole di controllo accesso NFS
  CIFS share     - Definizioni di share SMB
  igroup         - Gruppi di initiator host SAN per la mappatura dei LUN
  Utenti/ruoli   - Utenti locali, gruppi e ruoli RBAC con scope all'SVM
  DNS/NIS/LDAP   - Configurazione dei name service per l'SVM

Isolamento degli SVM:
  - Ogni SVM è completamente isolato dagli altri SVM sullo stesso cluster
  - Un SVM ha i propri indirizzi IP, configurazioni di protocollo e controlli di accesso
  - La compromissione o la configurazione errata di un SVM non influisce sugli altri


2.3 Data Path: dall'Host al Disco
-----------------------------------
Capire come i dati fluiscono da un host allo storage fisico aiuta a diagnosticare problemi
di performance e configurazione:

  Applicazione Host
       |
  OS Host / Filesystem
       |
  Protocollo Storage (NFS / SMB / iSCSI / FC / NVMe-oF)
       |
  Rete / SAN Fabric
       |
  ONTAP LIF (Logical Interface — IP o WWPN)
       |
  ONTAP Protocol Engine (server NFS, server CIFS, target iSCSI, target FC)
       |
  ONTAP Volume (FlexVol o FlexGroup)
       |
  ONTAP Write Cache (NVRAM — mirrored sul partner HA)
       | (scaricato su disco periodicamente o quando pieno)
  ONTAP WAFL (Write Anywhere File Layout) — motore filesystem
       |
  RAID Group (calcolo parità RAID-DP o RAID-TEC)
       |
  Drive fisico (HDD / SSD / NVMe)

WAFL — WRITE ANYWHERE FILE LAYOUT
WAFL è il motore filesystem interno di ONTAP, progettato specificamente per ottimizzare le
performance di scrittura e abilitare snapshot istantanei.

Caratteristiche chiave di WAFL:
  - Write anywhere: i nuovi dati vengono sempre scritti su blocchi liberi, mai in-place (nessun
    ciclo read-modify-write)
  - Copy-on-Write per gli snapshot: i blocchi esistenti non vengono mai sovrascritti — i dati
    modificati vanno su nuovi blocchi; i vecchi blocchi vengono preservati per gli snapshot
  - Consistency point (CP): WAFL raggruppa le scritture e le scarica periodicamente dalla NVRAM
    al disco in grandi scritture sequenziali (massimizzando le performance HDD e riducendo la
    write amplification degli SSD)
  - Integrazione NVRAM: tutte le scritture vengono prima committed alla NVRAM (persistente,
    mirrored sul partner HA), rendendole durevoli anche prima di raggiungere il disco


MODULE 3 - FUNZIONALITA' DI DATA PROTECTION IN ONTAP
======================================================

3.1 Panoramica della Data Protection ONTAP
--------------------------------------------
ONTAP fornisce un insieme completo e integrato di funzionalità di data protection che coprono
tutti i principali scenari di perdita di dati:

  Cancellazione accidentale file - Protezione: Snapshot + SnapRestore      - Recovery: Secondi
  Corruzione volume              - Protezione: Snapshot revert              - Recovery: Secondi-minuti
  Guasto drive                   - Protezione: RAID-DP/TEC + hot spare      - Recovery: Automatico, no perdita
  Guasto nodo                    - Protezione: HA pair takeover             - Recovery: Secondi
  Guasto sito                    - Protezione: MetroCluster/SnapMirror sync - Recovery: Secondi-minuti
  Attacco ransomware             - Protezione: ARP + Snapshot + SnapLock   - Recovery: Minuti (da ultimo snapshot pulito)
  Backup a lungo termine         - Protezione: SnapVault/SnapMirror Cloud  - Recovery: Ore (dipende da RTO)
  Retention compliance           - Protezione: SnapLock WORM               - N/A (garanzia di immutabilità)


3.2 Snapshot Copies
---------------------
Gli Snapshot ONTAP sono il meccanismo di data protection fondamentale su cui si basano tutte le
altre funzionalità di protezione ONTAP.

Come funzionano gli Snapshot in ONTAP:
Gli snapshot ONTAP sono implementati usando redirect-on-write (una variante di Copy-on-Write):
  - Quando viene preso uno snapshot, ONTAP registra un puntatore a ogni blocco attivo nel volume
  - Lo snapshot non copia i dati — cattura semplicemente la mappa dei blocchi corrente
  - Quando i dati attivi cambiano, i nuovi dati vengono scritti su un nuovo blocco (write
    anywhere)
  - Il vecchio blocco viene preservato e puntato dallo snapshot
  - Lo snapshot consuma spazio solo per i blocchi modificati dalla sua creazione

Consumo di spazio (esempio):
  Età snapshot:  1 ora    4 ore    24 ore    1 settimana
  Spazio usato:  ~0,5%    ~2%      ~5%       ~15%
  (varia molto in base al tasso di modifica dei dati)

Snapshot Policy:
ONTAP consente di configurare schedule automatici di snapshot con regole di retention:
  Orario      - Retention tipica: 6 copie   - Naming: hourly.YYYY-MM-DD_HHmm
  Giornaliero - Retention tipica: 2 copie   - Naming: daily.YYYY-MM-DD_0010
  Settimanale - Retention tipica: 2 copie   - Naming: weekly.YYYY-MM-DD_0015

Gli amministratori possono creare policy snapshot personalizzate per qualsiasi combinazione di
schedule e retention.

Accesso agli Snapshot:

Da client NAS:
  - Client NFS: snapshot visibili nella directory nascosta .snapshot alla radice di qualsiasi
    volume montato
  - Client SMB: snapshot accessibili tramite ~snapshot o funzionalità "Versioni precedenti" in
    Windows Explorer
  - Gli utenti finali possono ripristinare autonomamente versioni precedenti dei file senza
    coinvolgere l'amministratore

Da ONTAP:
  - volume snapshot restore-file: ripristina singoli file
  - volume snapshot restore: ripristina l'intero volume allo stato dello snapshot


3.3 SnapRestore
----------------
SnapRestore è la funzionalità ONTAP che abilita il recovery rapido ed efficiente dagli snapshot.

  Volume revert       - Ripristina un intero volume a uno stato snapshot precedente  - Secondi (solo operazione metadata)
  Single file/LUN     - Ripristina un file o LUN specifico da uno snapshot            - Secondi-minuti
  Partial file restore - Ripristina specifici byte range all'interno di un file       - Secondi

Nota importante sul Volume revert: il ripristino di un volume a uno snapshot è irreversibile — tutte
le modifiche effettuate dopo lo snapshot vengono scartate. ONTAP richiede conferma e raccomanda di
creare un nuovo snapshot prima del ripristino.


3.4 SnapMirror
---------------
SnapMirror replica gli snapshot ONTAP tra sistemi di storage per disaster recovery, distribuzione
dei dati e backup cloud.

Tipi di relazione SnapMirror:
  Async SnapMirror      - RPO: Minuti    - Replica snapshot su schedule; destinazione è volume DP read-only - Uso: DR, backup offload
  Sync SnapMirror       - RPO: Zero      - Ogni scrittura confermata su entrambi i sistemi prima dell'ACK  - Uso: DR senza perdita dati
  SnapMirror Active Sync - RPO+RTO: Zero - Entrambi i siti servono I/O; failover trasparente               - Uso: Business continuity
  SnapMirror Cloud      - RPO: Ore       - Replica verso object storage S3-compatible                      - Uso: Backup cloud, archivio

Workflow di replica SnapMirror:

Trasferimento baseline iniziale:
  1. Sorgente e destinazione stabiliscono una relazione SnapMirror
  2. Viene trasferito uno snapshot baseline completo (tutti i dati)
  3. Il volume di destinazione entra in uno stato DP read-only

Aggiornamenti incrementali:
  1. ONTAP crea un nuovo snapshot sulla sorgente
  2. Solo i blocchi modificati dall'ultimo snapshot comune vengono trasferiti
  3. La destinazione viene aggiornata per riflettere il nuovo stato
  4. Gli snapshot intermedi vengono eliminati secondo la policy di retention

Failover / Disaster Recovery:
  1. La sorgente diventa non disponibile
  2. L'amministratore interrompe la relazione SnapMirror sulla destinazione (snapmirror break)
  3. Il volume di destinazione diventa read-write
  4. Gli host vengono reindirizzati alla destinazione
  5. Quando la sorgente viene ripristinata: snapmirror resync — trasferisce solo i blocchi
     modificati


3.5 SnapVault
--------------
SnapVault è la soluzione di backup disk-to-disk di ONTAP, ottimizzata per la retention dei dati a
lungo termine indipendente dalla sorgente.

Differenze chiave rispetto a SnapMirror:
  Scopo primario      - SnapMirror: replica DR                    / SnapVault: backup/retention a lungo termine
  Destinazione        - SnapMirror: mirror della sorgente          / SnapVault: backup indipendente con propria retention
  Policy retention    - SnapMirror: coincide con la sorgente       / SnapVault: policy più lunga e indipendente
  Indipendenza snap   - SnapMirror: cancellare snap sorgente       / SnapVault: cancellare snap sorgente NON influisce
                        influisce sul mirror                         sul vault
  Relazione           - SnapMirror: volume-to-volume              / SnapVault: volume-to-volume (policy separata)

Uso tipico: mantieni 7 snapshot giornalieri sulla produzione, mantieni 52 settimanali + 24 mensili
sul secondario SnapVault.


3.6 MetroCluster
-----------------
MetroCluster fornisce alta disponibilità a livello di sito con mirroring sincrono dei dati tra due
cluster ONTAP geograficamente separati.

Architettura MetroCluster:
  Sito A                                    Sito B
  Cluster ONTAP A (Node 1, Node 2)          Cluster ONTAP B (Node 3, Node 4)
        |                                         |
        |<------- Mirror Sincrono -------------->|
        |  (ogni scrittura confermata             |
        |   su entrambi i siti prima dell'ACK)    |
        |                                         |
  ISL (Inter-Switch Link)               ISL
  Rete FC o IP                          Rete FC o IP

Proprietà chiave di MetroCluster:
  - RPO = 0: nessuna perdita di dati anche se un intero sito si guasta
  - RTO = ~30 secondi: switchover automatico in caso di guasto del sito
  - Entrambi i siti possono servire I/O di lettura e scrittura simultaneamente
  - Distanze supportate: fino a 300 km (FC) o 700 km (IP con amplificazione DWDM)

Componenti MetroCluster:
  ONTAP Mediator  - Witness terzo che risolve i pareggi durante le partizioni di rete; previene
                    lo split-brain
  ISL             - Connessione di rete tra i siti (fabric FC o rete IP)
  Switchover      - Sito A si guasta → Sito B prende in carico tutti i workload
  Switchback      - Sito A ripristinato → i workload tornano al Sito A


3.7 SnapLock
-------------
SnapLock fornisce storage WORM (Write Once, Read Many) — i dati scritti su un volume SnapLock non
possono essere modificati o cancellati fino alla scadenza del periodo di retention.

Modalità SnapLock:
  Compliance  - Chi può cancellare prima della scadenza: Nessuno (nemmeno l'admin) - Uso: Compliance normativa (SEC 17a-4, CFTC, FINRA, HIPAA)
  Enterprise  - Chi può cancellare prima della scadenza: Admin con privilegi speciali - Uso: Protezione ransomware, governance interna

SnapLock per la protezione dal ransomware:
Quando gli snapshot di backup SnapVault sono memorizzati su un volume SnapLock Enterprise:
  - Il ransomware non può cancellarli (anche con credenziali admin compromesse)
  - Il punto di recovery dal ransomware è garantito essere preservato
  - Gli amministratori mantengono un backup noto-buono per il ripristino

Periodi di retention:
  - Periodi di retention minimi e massimi possono essere definiti per volume SnapLock
  - I file possono avere date di retention individuali (file WORM)
  - Il compliance clock (gestito da ONTAP, indipendente dall'ora di sistema) previene la
    manomissione dei periodi di retention


3.8 Autonomous Ransomware Protection (ARP)
-------------------------------------------
ARP è il motore di rilevamento ransomware integrato AI-powered di ONTAP.

Pipeline di rilevamento ARP:
  Attività I/O del Volume
       |
  Fase di Apprendimento (30 giorni)
    --> Stabilire la baseline normale:
        - Tipi di file ed estensioni in uso
        - Livelli di entropia dei dati scritti
        - Pattern I/O (ora del giorno, velocità, dimensione)
       |
  Fase di Monitoraggio Attivo
    --> Analisi comportamentale continua
       |
  Trigger di Rilevamento Anomalie:
    +-- Nuove/sconosciute estensioni di file (firma di cifratura)
    +-- Scritture ad alta entropia (dati cifrati ~8,0 bit/byte di entropia)
    +-- Rinomina massiva di file (il ransomware rinomina i file dopo la cifratura)
    +-- Picco anomalo del tasso I/O (operazione di cifratura massiva)
       |
  Auto-risposta:
    +-- Crea snapshot immutabile immediatamente
    +-- Genera alert (email / SNMP / evento EMS)
    +-- L'admin esamina e conferma o contrassegna come falso positivo

Modalità ARP:
  Learning mode    - ARP osserva e costruisce una baseline comportamentale; nessun alert generato
  Active mode      - ARP applica il rilevamento e agisce sulle anomalie
  ARP/AI (9.15+)   - Modello AI potenziato; transizione più rapida; accuratezza migliorata


MODULE 4 - NETWORKING NEI CLUSTER ONTAP
=========================================

4.1 Panoramica dell'architettura di rete ONTAP
------------------------------------------------
I cluster ONTAP usano più reti distinte per scopi diversi. Separare queste reti migliora la
sicurezza, le performance e la semplicità di gestione.

  Rete Cluster       - Comunicazione privata inter-nodo (cluster interconnect)
  Rete Dati          - Accesso client (NFS, SMB, iSCSI, NVMe/TCP)
  Rete Gestione      - Accesso amministrativo (System Manager, CLI, API)
  Rete Intercluster  - Traffico SnapMirror e peering cluster
  SAN Fabric (FC)    - Percorso dati Fibre Channel (rete fisica separata)


4.2 Cluster Network (Cluster Interconnect)
-------------------------------------------
La rete cluster è una rete privata ad alta velocità dedicata che connette tutti i nodi all'interno
di un cluster. Questa rete non viene mai usata per il traffico dati client.

Scopo:
  - Comunicazione inter-nodo (coordinazione cluster, quorum)
  - Mirroring NVRAM tra nodi dell'HA pair
  - Spostamento dati tra nodi (spostamenti di volume, bilanciamento del carico)
  - Traffico di gestione del cluster

Requisiti:
  - Switch dedicati (switch cluster interconnect) — non condivisi con reti dati o di gestione
  - Velocità tipiche: 10 GbE, 25 GbE, 40 GbE, o 100 GbE
  - La bassa latenza è critica
  - Ogni nodo ha un minimo di due porte cluster per ridondanza (active-active)

Cluster LIF:
  - ONTAP crea automaticamente Cluster LIF sulle porte cluster di ogni nodo
  - Le Cluster LIF usano indirizzi IP link-local (169.254.x.x) — non instradabili esternamente


4.3 Rete Dati
--------------

LOGICAL INTERFACE (LIF)
Una LIF (Logical Interface) è un endpoint di rete virtuale — un indirizzo IP o un FC WWPN — tramite
cui i client accedono ai dati. Le LIF sono l'astrazione chiave che abilita la mobilità dei dati in
ONTAP.

Proprietà delle LIF:
  Indirizzo IP / WWPN - L'identità di rete della LIF
  Home port           - La porta fisica dove la LIF normalmente risiede
  Current port        - La porta fisica dove la LIF è attualmente attiva (può differire
                        dall'home durante il failover)
  Home node           - Il nodo dove la LIF normalmente risiede
  Current node        - Il nodo dove la LIF è attualmente attiva
  SVM                 - L'SVM che possiede questa LIF
  Protocollo          - I protocolli serviti da questa LIF (NFS, SMB, iSCSI, NVMe/TCP, FC, ecc.)

Tipi di LIF:
  Data LIF          - NFS, SMB, iSCSI, NVMe/TCP, S3  - Accesso dati client
  FC LIF (FC Target) - Fibre Channel, FCoE           - Accesso SAN FC
  Cluster LIF       - Interno                         - Cluster interconnect (auto-creata)
  Node mgmt LIF     - HTTPS, SSH                      - Accesso gestione per nodo
  Cluster mgmt LIF  - HTTPS, SSH                      - Accesso gestione a livello cluster
  Intercluster LIF  - SnapMirror                      - Peering cluster e replica

LIF FAILOVER
Il LIF failover garantisce che la connettività client venga mantenuta anche in caso di guasto di una
porta fisica o di un nodo.

Policy di failover:
  system-defined        - La LIF può migrare su qualsiasi porta nello stesso broadcast domain
  local-only            - La LIF può migrare solo su porte nello stesso nodo
  sfo-partner-only      - La LIF può migrare solo sul nodo partner SFO (storage failover)
  broadcast-domain-wide - La LIF può migrare su qualsiasi porta nel broadcast domain (qualsiasi nodo)
  disabled              - La LIF non fa failover (non raccomandato per le data LIF)

Failover automatico:
  1. La porta fisica sul Node 1 si guasta
  2. ONTAP rileva l'evento link-down
  3. La LIF migra automaticamente su una porta sana (sullo stesso nodo o sul nodo partner)
  4. Aggiornamento DNS (se configurato) reindirizza i client al nuovo IP se cambiato
  5. Quando la porta originale si ripristina, la LIF torna alla home port (auto-revert)


4.4 Componenti e concetti di rete
-----------------------------------

BROADCAST DOMAIN
Un broadcast domain in ONTAP è un raggruppamento logico di porte di rete che condividono lo stesso
segmento di rete Layer 2 (la stessa VLAN e subnet). ONTAP usa i broadcast domain per determinare i
target di failover validi per le LIF. ONTAP scopre automaticamente l'appartenenza ai broadcast
domain usando LLDP/CDP.

IPSPACE
Un IPspace è un namespace di rete logico che contiene uno o più broadcast domain.
  - L'IPspace Default contiene tutte le reti dati e di gestione standard
  - IPspace aggiuntivi forniscono isolamento di rete per ambienti multi-tenant (ogni SVM in
    un IPspace diverso ha una rete completamente isolata)
  - Gli SVM in IPspace diversi possono avere indirizzi IP sovrapposti senza conflitti

VLAN
ONTAP supporta il VLAN tagging 802.1Q sulle porte di rete fisiche:
  - Crea interfacce VLAN per segmentare il traffico su infrastruttura fisica condivisa
  - Esempio: crea VLAN 100 sulla porta e0c → nuova porta logica e0c-100
  - Assegna protocolli diversi a VLAN diverse per la separazione del traffico

LINK AGGREGATION (LACP / ifgroup)
ONTAP supporta gli interface group (ifgroup) — equivalente del link aggregation / bonding:

  LACP (802.3ad)  - Aggregazione link dinamica; coordinazione con lo switch richiesta - Standard per la maggior parte dei deployment
  Static          - Aggregazione manuale senza negoziazione LACP - Quando lo switch non supporta LACP
  Single-mode     - Active/standby (non un vero aggregato) - Semplice failover

Vantaggi degli ifgroup:
  - Banda aumentata (più link in parallelo)
  - Ridondanza (se un link si guasta, gli altri continuano)
  - Load balancing tra i link


4.5 Protocolli Storage e requisiti di rete
-------------------------------------------

NFS (NETWORK FILE SYSTEM)
  - Trasporto: TCP (o UDP per client NFSv3 datati)
  - Porta: TCP 2049
  - Requisito ONTAP: Data LIF con protocollo NFS abilitato sull'SVM
  - Controllo accesso client: le export policy definiscono quali client possono montare quali
    volumi
  - Kerberos: supportato per la sicurezza NFSv4 (SEC_KRB5, SEC_KRB5I, SEC_KRB5P)

SMB/CIFS (WINDOWS FILE SHARING)
  - Trasporto: TCP
  - Porta: TCP 445
  - Requisito ONTAP: server CIFS unito a un dominio Active Directory (o workgroup)
  - Autenticazione: Active Directory Kerberos (raccomandato) o NTLM
  - Controllo accesso: permessi a livello di share + ACL NTFS sui file

iSCSI (IP SAN)
  - Trasporto: TCP
  - Porta: TCP 3260
  - Requisito ONTAP: servizio iSCSI abilitato sull'SVM; LIF iSCSI creata
  - Requisito host: iSCSI initiator (software o HBA hardware)
  - Discovery: iSCSI Discovery (send targets) o configurazione manuale del target
  - Multipath: MPIO/DM-Multipath per ridondanza e load balancing

NVME/TCP
  - Trasporto: TCP
  - Porte: TCP 8009 (discovery), TCP 4420 (I/O)
  - Requisito ONTAP: servizio NVMe abilitato; LIF NVMe sul data SVM
  - Requisito host: software NVMe/TCP e HBA per l'host
  - Performance: latenza inferiore a iSCSI; adatto per workload latency-sensitive

FIBRE CHANNEL (FC)
  - Trasporto: Fabric FC dedicato (rete fisica separata)
  - Requisito ONTAP: adapter FC target (HBA) nel nodo; LIF FC configurate
  - Zoning: lo zoning dello switch FC limita quali host possono vedere quali porte storage
  - Multipath: MPIO o DM-Multipath per ridondanza

5.2 Monitoraggio ONTAP
------------------------

EMS — EVENT MANAGEMENT SYSTEM
L'EMS (Event Management System) è il motore di notifica interno di ONTAP. Genera eventi per
praticamente ogni operazione o condizione significativa nel sistema.

Livelli di severità degli eventi EMS:
  EMERGENCY     - Sistema non utilizzabile; azione immediata richiesta - Es: nodo giù, aggregate offline
  ALERT         - Condizione critica che richiede attenzione immediata - Es: guasto drive, lag SnapMirror superato
  ERROR         - Condizione di errore che può impattare la funzionalità - Es: LUN offline, creazione snapshot fallita
  WARNING       - Potenziale problema; azione raccomandata             - Es: aggregate quasi pieno (>90%)
  NOTICE        - Condizione normale ma significativa                  - Es: takeover nodo avviato, volume online
  INFORMATIONAL - Eventi operativi normali                            - Es: snapshot creato, volume ridimensionato
  DEBUG         - Informazioni diagnostiche                           - Cambiamenti di stato interni (normalmente nascosti)

Destinazioni di notifica EMS:
  - Email: invia alert alle distribution list del team on-call
  - SNMP trap: integra con piattaforme di monitoring SNMP (Nagios, PRTG, SolarWinds)
  - Syslog: inoltra eventi a sistemi SIEM (Splunk, QRadar, Elastic SIEM)
  - REST webhook: invia eventi a endpoint HTTP personalizzati (Slack, PagerDuty, ServiceNow)

AUTOSUPPORT
AutoSupport è il meccanismo di notifica al supporto automatizzato di ONTAP. Invia messaggi periodici
e attivati da eventi al supporto NetApp e opzionalmente al team IT del cliente.

Tipi di messaggi AutoSupport:
  Weekly          - Ogni domenica a mezzanotte - Configurazione completa sistema e statistiche
  Daily           - Ogni giorno                - Statistiche performance e riepilogo salute
  Event-triggered - Su condizioni specifiche   - Panic, core dump, drive guasto, EMS alta severità
  Manual          - Avviato dall'admin         - On-demand per casi di supporto

Metodi di consegna AutoSupport:
  - HTTPS: metodo di consegna primario (porta 443)
  - SMTP: consegna via email (alternativa)

Vantaggi AutoSupport:
  - Il supporto NetApp può identificare proattivamente i problemi prima che causino interruzioni
  - I dati AutoSupport alimentano le analytics di Active IQ
  - Richiesto per alcuni diritti di supporto NetApp

5.3 Attività di manutenzione ONTAP
------------------------------------

AGGIORNAMENTI SOFTWARE E FIRMWARE

Aggiornamenti software ONTAP:
  - Rolling upgrade: aggiorna un nodo alla volta; il cluster rimane online
  - Ogni HA pair subisce: takeover → upgrade → giveback → si ripete per il prossimo pair
  - I client subiscono una breve pausa I/O (<30 secondi) durante takeover/giveback

Workflow di upgrade raccomandato:
  1. Revisiona Upgrade Advisor di Active IQ per la versione raccomandata e i rischi
  2. Scarica l'immagine software ONTAP dal sito di supporto NetApp
  3. Carica l'immagine nel cluster: system image get
  4. Valida l'immagine: system image show
  5. Esegui rolling upgrade automatizzato: system node image update-auto
  6. Verifica il completamento: system node image show

Aggiornamenti firmware disco:
  - Il firmware dei dischi è incluso nelle release ONTAP
  - Aggiornato automaticamente durante gli upgrade ONTAP (non distruttivo)
  - Può anche essere attivato manualmente: storage disk firmware update

GESTIONE DELLA CAPACITA'
Attività chiave di monitoraggio della capacità:

Capacità degli aggregate:
  storage aggregate show -fields size,used,available,percent-used
  Best practice: alert quando l'aggregate raggiunge l'80% di capacità; pianifica l'espansione
  prima del 90%.

Capacità dei volumi:
  volume show -fields size,used,available,percent-used
  Monitora attentamente i volumi thin-provisioned — possono esaurire lo spazio improvvisamente.

Capacità degli snapshot:
  volume snapshot show -fields size
  volume show -fields snapshot-count,snapshot-space-used
  Grandi numeri di snapshot vecchi possono consumare spazio significativo; rivedi regolarmente
  le policy di retention.

Capacità FabricPool:
  storage aggregate object-store show
  volume show -fields tiering-policy,cloud-retrieval-policy

GESTIONE UTENTI E ACCESSI

ONTAP RBAC (Role-Based Access Control):

Ruoli integrati:
  admin           - Amministrazione completa del cluster
  backup          - Accesso read-only + operazioni di backup
  readonly        - Accesso read-only a tutti gli oggetti
  vsadmin         - Amministratore SVM (con scope a un SVM)
  vsadmin-readonly - Accesso SVM read-only
  vsadmin-backup  - Operazioni di backup SVM

Ruoli personalizzati:
  security login role create -role vol_manager -cmddirname volume -access all
  security login role create -role vol_manager -cmddirname storage -access readonly

Metodi di autenticazione utente:
  Password         - Password ONTAP locale
  SSH public key   - Autenticazione SSH basata su chiave
  Active Directory - Autenticazione di dominio tramite AD
  LDAP             - Autenticazione tramite directory LDAP
  SAML             - SAML 2.0 per UI web System Manager (ONTAP 9.3+)
  Certificate      - Certificato X.509 per accesso API

Multi-Admin Verification (MAV):
  security multi-admin-verify modify -approval-groups admin_group -required-approvers 1
  security multi-admin-verify rule create -operation "volume snapshot delete"
  security multi-admin-verify rule create -operation "snapmirror delete"

HEALTH CHECK COMUNI
Una routine di health check giornaliera/settimanale per gli amministratori ONTAP:

  1.  cluster show / system health status show        -- Salute del cluster
  2.  system node show / storage failover show         -- Stato nodi
  3.  storage aggregate show / show-status             -- Salute e capacità aggregate
  4.  volume show -state !online                       -- Volumi non online
      volume show -fields percent-used                 -- Capacità volumi
  5.  snapmirror show -health false                    -- Relazioni SnapMirror non sane
      snapmirror show -fields lag-time                 -- Lag di replica
  6.  storage disk show -broken                        -- Dischi guasti
      storage disk show -container-type spare          -- Verifica disponibilità spare
  7.  network interface show -status-oper down         -- LIF non attive
      network port show -health-status degraded        -- Porte degradate
  8.  event log show -severity emergency,alert -time-range 24h  -- Eventi EMS recenti
  9.  job show -state running,queued                   -- Job in corso
  10. system node autosupport show                     -- Stato AutoSupport

ONTAP CLUSTER FUNDAMENTALS
|
+-- UNDERSTANDING ONTAP CLUSTERS
|   +-- Cluster: 1-24 nodi, unico sistema logico
|   +-- Node: singolo storage controller (CPU, RAM, drive, NVRAM)
|   +-- HA Pair: 2 nodi, NVRAM mirrored, takeover/giveback reciproco
|   |   +-- Takeover (failover) --> Giveback (ripristino)
|   +-- Quorum: meccanismo di consistenza cluster; epsilon per tie-breaking
|
+-- ONTAP STORAGE ARCHITECTURE
|   +-- Fisico: Drive (HDD/SSD/NVMe) --> Disk Shelf
|   +-- RAID: RAID-DP (2 parità) - RAID-TEC (3 parità) + Hot Spare
|   +-- Aggregate: RAID group per nodo; root aggr + data aggr
|   +-- Volume: FlexVol (300TB) - FlexGroup (20PB) - SnapLock - DP
|   +-- Qtree: partizione directory con quota / export policy
|   +-- LUN: oggetto block storage (iSCSI / FC), mappato tramite igroup
|   +-- SVM: tenant logico (proprie LIF, protocolli, volumi, sicurezza)
|   +-- WAFL: Write Anywhere File Layout; snapshot CoW; NVRAM CP
|
+-- DATA PROTECTION
|   +-- Snapshot: CoW istantaneo, ripristino self-service, fino a 1023/vol
|   +-- SnapRestore: volume revert o ripristino singolo file/LUN (secondi)
|   +-- SnapMirror: async - sync - active sync - cloud
|   +-- SnapVault: retention backup a lungo termine indipendente
|   +-- MetroCluster: mirror sincrono, RPO=0, RTO~30s, fino a 700km
|   +-- SnapLock: WORM, Compliance (inviolabile) vs Enterprise
|   +-- ARP: rilevamento comportamentale AI --> snapshot automatico --> alert
|
+-- NETWORKING NEI CLUSTER ONTAP
|   +-- Cluster Network: inter-nodo privata (mirror NVRAM, quorum)
|   +-- Data Network: I/O client (NFS, SMB, iSCSI, NVMe/TCP, FC)
|   +-- Management Network: accesso admin (SSH, HTTPS, API)
|   +-- Intercluster Network: SnapMirror, peering cluster
|   +-- LIF: endpoint rete virtuale; home/current port; policy failover
|   +-- Broadcast Domain: raggruppamento L2 porte (confine failover)
|   +-- IPspace: namespace rete per multi-tenancy
|   +-- VLAN: tagging 802.1Q per segmentazione traffico
|   +-- ifgroup (LACP): link aggregation per banda e ridondanza
|
+-- GESTIONE SISTEMA ONTAP
    +-- System Manager: GUI web, dashboard, workflow guidati
    +-- CLI: SSH, comandi gerarchici, scripting
    +-- REST API: JSON/REST, Swagger docs, Python/Ansible/Terraform
    +-- BlueXP: piano di gestione hybrid cloud unificato
    +-- Active IQ: advisory AI-driven, alert rischi, guida upgrade
    +-- EMS: livelli severità eventi (Emergency-->Debug); email/SNMP/syslog
    +-- AutoSupport: telemetria automatizzata al supporto NetApp
    +-- Performance: grafici System Manager, CLI statistiche, OPM
    +-- Manutenzione: rolling ONTAP upgrade, controlli capacità, RBAC, MAV
